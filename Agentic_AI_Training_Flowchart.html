Sequence Models — NLP with Neural Networks
How neural networks were applied to NLP before Transformers, and the breakthrough that changed everything.
RNN for NLP
Reads text word by word Hidden state carries context Bidirectional RNN (reads both ways) Problem: forgets distant words
Use for: Sentiment, basic classification, POS tagging
Seq2Seq — Encoder–Decoder
Encoder compresses full input into one vector Decoder generates output from that vector Machine translation (English → French) Problem: one vector loses information
Use for: Translation, summarisation (pre-Transformer)
Attention Mechanism — the breakthrough
Decoder looks at ALL encoder states Query (Q) — what am I looking for? Key (K) — what does each word offer? Value (V) — what to actually retrieve Score = Q · Kᵀ / √d → Softmax Output = weighted sum of Values
"Attention is all you need" (2017) → Led to Transformers
drop the RNN entirely — build everything with attention — 2017 paper
BLOCK 05
Transformer Architecture
"Attention Is All You Need" (2017, Google) — replaced RNNs entirely with self-attention. Fully parallelisable so it scales to billions of parameters on GPUs. Became the backbone of every modern AI model.
Foundation of all modern LLMs
One Transformer Block — repeated N times to create depth
Input preparation
Token → Input Embedding (word to vector) Positional Encoding — inject word order (sin/cos or learned)
Attention sublayer
Multi-Head Self-Attention — each token attends to every other Multiple heads = multiple relationship perspectives Add & LayerNorm — residual connection stabilises training
Feed-forward sublayer
Feed-Forward Network (FFN) — processes each token independently Add & LayerNorm — second residual connection Repeat this whole block × N layers
Why residual connections?
Add input back to output at each sublayer Gradients flow cleanly through deep networks Enables very deep stacking (24–96+ layers)
Three Transformer Variants
Encoder (BERT-style)
Reads the full sentence in both directions simultaneously — builds deep contextual understanding.
Structure
Bidirectional — sees full input N stacked encoder blocks No causal mask
Pre-training task
Masked LM — predict masked words Next Sentence Prediction
Use for
Text classification NER & tagging Question answering Sentence embeddings
Models: BERT, RoBERTa
Decoder (GPT-style)
Generates text left-to-right, one token at a time. Can only see previous tokens — never future ones.
Structure
Autoregressive — generates left to right Causal mask — can't see future tokens N stacked decoder blocks
Pre-training task
Next token prediction (CLM)
Use for
Text generation Chatbots Code generation Creative writing
Models: GPT-2/3/4, LLaMA, Claude
Encoder–Decoder (T5-style)
Encoder reads and understands the input; Decoder generates the output. Best for tasks that transform one sequence into another.
Structure
Encoder reads full input Cross-attention connects encoder ↔ decoder Decoder generates output
Use for
Machine translation Text summarisation Question answering Any sequence-to-sequence task
Models: T5, BART
scale a decoder Transformer on trillions of tokens → LLM

BLOCK 10
RAG — Retrieval-Augmented Generation
LLMs know what they were trained on — nothing more. RAG gives them access to fresh, private, or specialised knowledge by retrieving relevant documents and injecting them into the context at query time. The most important pattern in real-world LLM applications.
Libraries: LangChain · LlamaIndex · ChromaDB · FAISS · Pinecone
Why RAG? — Problems it solves
Knowledge cutoff — LLM doesn't know recent events → retrieve live documents Private data — LLM wasn't trained on your company docs → retrieve them Hallucination — LLM makes things up → ground it in real retrieved facts Long documents — LLM can't read 1000 pages → retrieve only relevant chunks Cost — cheaper than fine-tuning for knowledge updates
RAG Pipeline — 8 steps from document to answer
Complete RAG Pipeline
Indexing phase — done once, offline
StepWhat happens
1. Document ingestionLoad PDFs, Word docs, web pages, databases
2. ChunkingSplit docs into pieces — fixed-size, sentence, or semantic
3. EmbeddingConvert each chunk to a vector using an embedding model
4. Vector storeSave all chunk vectors in a searchable vector database
Retrieval + Generation phase — at query time
StepWhat happens
5. Query embeddingConvert user's question to a vector using same model
6. Similarity searchFind top-K chunks whose vectors are closest to query
7. Context injectionInsert retrieved chunks into the LLM prompt
8. LLM generationModel answers using the retrieved context as grounding
Key Concepts
ConceptPurpose
Chunking strategiesFixed-size / sentence / semantic / recursive character
Embedding modelsConvert text to vectors — text-embedding-3, BGE, E5
Vector databasesFAISS (local fast), Chroma (easy), Pinecone (cloud)
Cosine similarityMeasure how similar query and chunk vectors are
Top-K retrievalReturn the K most relevant chunks
RerankingSecond pass to improve chunk relevance ordering
Hybrid searchCombine keyword search + semantic search for better recall
RAG Typesconcept level
TypeWhat it is
Naive RAGBasic pipeline — retrieve → inject → generate. Start here.
Advanced RAGBetter chunking, reranking, query rewriting for more accuracy
Modular RAGSwap out pipeline components like Lego blocks
Agentic RAGAgent decides when and what to retrieve dynamically
Libraries for RAG
LangChain — end-to-end RAG pipeline LlamaIndex — document-focused RAG ChromaDB — lightweight local vector store FAISS — fast vector search Pinecone — managed cloud vector DB HuggingFace — embedding models

BLOCK 11
Context Engineering
The evolution beyond prompt engineering — not just how you phrase the request, but what information you place in the context window, how you structure it, and how you manage what the model can see at any given moment. The model can only reason about what it can see.
Libraries: LangChain Memory · LlamaIndex · HuggingFace
What is the Context Window?
ConceptWhat it means
Context windowMaximum text the model can process in one call
TokensHow text is counted — roughly 1 word ≈ 1.3 tokens
Context limit growth4K → 32K → 128K → 1M+ tokens over time
What fits insideSystem prompt + conversation + documents + examples
Lost-in-the-middle problem
Models pay most attention to start and end of context Important info buried in the middle gets ignored Solution: put key instructions first and repeat at end
What context engineering manages
What documents to include How much conversation history to keep Where to place examples and instructions How to compress when limit is reached

BLOCK 13
Platforms
Platforms provide the infrastructure to access, deploy, and scale LLM applications — without managing GPU servers yourself. Three major cloud providers offer managed LLM services. You write the application; the platform handles compute, scaling, and model access.
