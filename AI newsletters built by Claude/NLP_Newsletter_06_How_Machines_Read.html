From the editor

Welcome to Issue 06 — Block 04. We've gone from gradient descent to neural networks. Now we turn those networks toward the hardest data type in existence: human language.

Language is not like images or tabular data. An image has a fixed spatial structure. A table has fixed columns. Language is fluid, ambiguous, context-dependent, and endlessly creative. "The deal is done" means something completely different in a business negotiation versus a magical realism novel. "Can you pass the salt?" is technically a question about ability but functionally a request. "This is fine" is often the opposite of fine.

NLP is the field that teaches machines to navigate all of this. And since every AI agent you'll ever build — whether it processes documents, answers questions, routes requests, or generates responses — works with language, NLP is the invisible foundation of everything in your roadmap from here forward.

This issue covers how machines read: cleaning raw text, the four layers of language understanding, and the twelve real tasks you can build with them.

Issue 07 covers how machines represent language as numbers — embeddings, Word2Vec, and the Attention mechanism that changed everything in 2017.

Part 1 · Why NLP is Hard
The problem first

Why language is the hardest data type in machine learning

Every other data type we've covered had a fixed structure. A 28×28 image is always 784 numbers. A customer record with 10 features is always 10 numbers. Neural networks love fixed structure — you know exactly what you're getting.

Language has none of this. Sentences are different lengths. Words can mean different things in different contexts. The same idea can be expressed in thousands of different ways. Grammar rules have exceptions to their exceptions. And meaning isn't in individual words — it's in the relationships between them.

Four sentences, same meaning

"The cat sat on the mat." / "On the mat sat the cat." / "The mat had a cat sitting on it." / "A cat was seated upon the mat." — Any NLP system must recognise these as semantically equivalent despite having completely different word orders and structures. A pixel-based system would never confuse four different images this way. Language does this constantly.

The core challenges every NLP system must solve: Ambiguity (bank = financial or river?), Context dependency (pronouns — "John told Mike that he was wrong" — who is he?), Variable length (text can be 3 words or 300,000 words), Compositionality (meaning of a sentence ≠ sum of its word meanings), and World knowledge ("Paris is the capital of France" is a fact the model must somehow know).

The NLP pipeline — always in this order

Every NLP system — from spam filters to GPT-4 — follows the same pipeline. Raw text goes in. It gets cleaned and preprocessed. It gets converted into numbers. A model processes those numbers. An output comes out. This issue covers steps 1–2 (cleaning + understanding). Issue 07 covers step 3 (text → numbers) and the neural models built on top.

Part 2 · Text Preprocessing
Always step one

Text Preprocessing — turning raw mess into clean input

Raw text from the real world is filthy. User reviews have typos, HTML tags, inconsistent capitalisation, emojis, and slang. News articles have publication dates, author bylines, and navigation text mixed with the actual content. Social media posts have hashtags, @mentions, and URLs. Before any model touches text, you clean it.

Think of preprocessing as standardising your data — the same reason you scale numerical features before training a neural network. You're making the data consistent so the model doesn't have to learn that "Running", "running", and "RUNNING" are the same thing.

Stemming vs Lemmatisation — the important difference

Stemming Crudely chop off word endings using rules. "Running" → "run". "Better" → "better". "Studies" → "studi". Fast and simple, but produces non-words ("studi" is not a word). Good enough for simple text classification tasks where you just need to group similar words, not for anything requiring real language understanding.
Lemmatisation Return the proper dictionary root form (the lemma) using linguistic knowledge. "Running" → "run". "Better" → "good". "Mice" → "mouse". Always produces real words. Slower than stemming but more accurate. Requires knowing the part of speech — "better" as an adjective lemmatises to "good", as a verb to "better". Use this when output quality matters.
Tokenisation Splitting text into individual units called tokens. The most basic tokenisation splits on whitespace and punctuation — producing words. But modern LLMs use subword tokenisation (BPE, WordPiece) where "unhappiness" becomes ["un", "happi", "ness"] — each piece a token. This handles rare words, misspellings, and multiple languages with a fixed vocabulary. Covered in detail in the next NLP issue.
Stop Words High-frequency words that carry little semantic meaning: "the", "a", "is", "are", "in", "on", "and", "or", "not". Removing them reduces noise and vocabulary size for classical NLP methods. But modern neural networks don't remove stop words — they can learn context from them. "The President" vs "A President" — the article matters.
When to preprocess — and when not to

Classical NLP (Bag of Words, TF-IDF, traditional ML classifiers) → always preprocess. The preprocessing removes noise that these simple models can't handle. Modern deep learning and LLMs → often skip preprocessing. BERT and GPT-4 are trained on raw text including punctuation, stop words, and capitalisation — they use all of it. Removing stop words before feeding text to a Transformer will actually hurt performance.

Part 3 · The Four Levels
How NLP understands language

Four levels — from letters to intent

NLP doesn't understand language in one shot. It builds understanding in layers — each level answering a more complex question about the text, each depending on the one below it.

The reading analogy

When you read "Elon Musk said Tesla would miss its quarterly targets, sending the stock down 8%", you do four things automatically: you identify the words (lexical). You parse that "Elon Musk said" is the main clause and "sending the stock down 8%" is a participial phrase (syntactic). You understand that Elon Musk is a person, Tesla is a company, and "stock down 8%" means a price drop (semantic). You infer that this is bad news for Tesla investors (pragmatic). NLP systems replicate each of these layers explicitly.

Part 4 · Level 1 — Lexical
Level 1 of 4
Lexical Analysis — What Are the Words?
The first layer of NLP. Identify and classify individual words, their forms, and basic properties. Everything else depends on getting this right.
Question: What words exist in this text?
Library: spaCy · NLTK
Level 1 · Lexical Analysis

Lexical — the alphabet of NLP

Lexical analysis is NLP's first pass at text. Before you can understand meaning, you need to know what units of language you're working with. This level breaks text into its components and labels their basic properties.

Tokenisation — the first cut

Every NLP pipeline starts with tokenisation. You can't process "I love NLP" as a string — you need the individual units. Word tokenisation gives you ["I", "love", "NLP"]. Sentence tokenisation splits a paragraph into sentences. Subword tokenisation (used in LLMs) breaks words into pieces.

POS Tagging Part-of-Speech tagging labels every word with its grammatical role: noun (NN), verb (VB), adjective (JJ), adverb (RB), determiner (DT), etc. "The quick brown fox jumps" → [DT, JJ, JJ, NN, VBZ]. Why it matters: the same word can be different parts of speech. "Time flies like an arrow" — is "flies" a noun (insects) or a verb (to fly)? POS tagging resolves this using context. Used as input to parsers, named entity recognisers, and sentiment systems.
Morphological Analysis Studying the internal structure of words — prefixes, suffixes, roots, and how they combine to create meaning. "Unhappiness" = "un" (negation prefix) + "happy" (root) + "ness" (state suffix). This matters for languages with complex word forms (German, Finnish, Turkish) and for handling words the model has never seen — if you understand "un" means negation and "happi" means joyful, you can infer "unhappiness" even without training data for that exact word.

Named Entity Recognition (NER) — finding proper nouns that matter

NER identifies and classifies named entities in text — real-world objects like people, organisations, locations, dates, and monetary values. It's one of the most used NLP capabilities in production systems.

NER in action — entities highlighted by type

When an agent processes customer emails, NER extracts product names, order numbers, and locations. In RAG pipelines, NER enriches document metadata so retrieval can filter by "documents mentioning Company X" or "documents from Q3 2024."

Library: spaCy · NLTK
Level 2 · Syntactic Analysis

Syntactic — grammar as a map of meaning

Knowing what words exist isn't enough. "Dog bites man" and "Man bites dog" use identical words but mean opposite things. The relationships between words — who is doing what to whom — is syntax. Syntactic analysis maps those relationships.

Parsing Analysing the grammatical structure of a sentence and representing it as a tree. There are two main types: Constituency parsing and Dependency parsing. Parsing reveals the skeletal structure of a sentence — what is the subject, what is the verb, what is the object, and how they relate. Used in machine translation (structure differs between languages), question answering (identifying the question's subject), and information extraction.
Dependency Parsing Draws directed arrows between words showing which word depends on which. In "The big dog bit the small cat" — "dog" is the subject of "bit", "cat" is the object of "bit", "big" modifies "dog", "small" modifies "cat". Each arrow is labelled with the relationship type (nsubj = nominal subject, dobj = direct object, amod = adjectival modifier). Modern NLP uses dependency parsing for information extraction and relation detection.
Constituency Parsing Breaks a sentence into nested phrase groups. "The quick fox" is a Noun Phrase (NP). "jumped over the fence" is a Verb Phrase (VP). NPs and VPs combine into sentences (S). The result is a hierarchical tree that shows how phrases group together. "The quick fox jumped over the fence" → S[NP[The quick fox] VP[jumped PP[over NP[the fence]]]]. More linguistically motivated than dependency parsing.
Context-Free Grammar (CFG) A formal rule system that defines how valid sentences are structured. Rules like: S → NP VP (a sentence is a noun phrase followed by a verb phrase), NP → DT NN (a noun phrase is a determiner followed by a noun), VP → VBD NP (a verb phrase is a past-tense verb followed by a noun phrase). CFGs are the mathematical backbone of constituency parsers. The "context-free" means the rules apply regardless of surrounding context — which is both their strength (simplicity) and limitation (real language is context-sensitive).
Does syntactic analysis still matter in the LLM era?

Transformers like BERT implicitly learn syntactic structure from data without explicit parsing. But syntax still matters in production for specific tasks: extracting who did what to whom in legal documents (dependency parsing), validating that generated SQL follows correct grammar (CFG), understanding negation scope ("The patient does not have a fever" — which entity is being negated?), and rule-based information extraction where you need precision guarantees that neural models don't provide.

Question: What is the meaning of this text?
Library: spaCy · HuggingFace Transformers
Level 3 · Semantic Analysis

Semantic — understanding actual meaning

Syntax tells you that "dog" is the subject and "bites" is the verb. Semantics tells you what biting means, who the dog is, and whether this is literally or figuratively true. This is the layer where NLP gets genuinely hard — and genuinely powerful.

Word Sense Disambiguation — one word, multiple meanings

English is full of words with multiple meanings. "Bank" can be a financial institution, a river bank, or to tilt an aircraft. "Light" can mean illumination, low weight, or pale colour. The same word in different contexts means fundamentally different things, and any model must resolve this to understand text correctly.

The disambiguation problem

"I went to the bank" — financial institution or river bank? Without context, impossible to know. "I went to the bank to deposit my cheque" → financial. "I went to the bank to fish" → river. "I sat on the bank and watched the water flow" → river (water flowing = strong context signal). Models learn to use surrounding words as signals to select the correct sense. This is word sense disambiguation — and it's one of the oldest, hardest problems in NLP.

Semantic Similarity A measure of how similar two pieces of text are in meaning — not in exact words, but in what they're about. "The car broke down" and "The vehicle stopped working" are semantically very similar despite sharing no content words. Computed by comparing embedding vectors (cosine similarity). Powers plagiarism detection, FAQ matching, duplicate question detection, and the retrieval step in RAG pipelines — find documents that mean the same thing as the user's query.
Sentiment Analysis Classify text as positive, negative, or neutral — or on a continuous scale. "This product is amazing" → positive. "Terrible experience, never again" → negative. More advanced: aspect-based sentiment analysis, which identifies sentiment toward specific entities. "The food was great but the service was horrible" → food: positive, service: negative. Used in brand monitoring, customer feedback analysis, financial news sentiment, and product review mining.
Coreference Resolution Identify when different expressions in a text refer to the same real-world entity. "John told Mike that he was wrong" — the pronoun "he" must be linked to either John or Mike. "The President signed the bill. She later addressed Congress" — "She" refers to "The President". This is crucial for understanding who did what in long documents. Without coreference resolution, extracting information about specific people or organisations from multi-sentence text is unreliable.
Relation Extraction Identify structured relationships between entities in text. "Apple was founded by Steve Jobs in 1976 in Cupertino" → (Apple, founded_by, Steve Jobs), (Apple, founded_in, 1976), (Apple, founded_at, Cupertino). These structured triples are used to build knowledge graphs — large networks of entities and their relationships. Google's Knowledge Graph (the box that appears with facts when you search) is built from relation extraction at massive scale.
Why semantic analysis is the heart of AI agents

When an AI agent receives "Book me a flight to Delhi for next Tuesday, I prefer window seats and no layovers", it must perform semantic analysis: extract entities (Delhi, next Tuesday), understand attributes (window seat, no layovers), resolve "next Tuesday" relative to today's date (coreference-like temporal reasoning), and understand that "book" means perform a transaction, not write a novel. Every agent instruction goes through semantic analysis before any action is taken.

Level 4 · Pragmatic Analysis

Pragmatic — what language actually does

This is where NLP gets closest to how humans actually use language. Semantics tells you what a sentence means literally. Pragmatics tells you what the speaker means by it — which is often completely different.

"Can you pass the salt?" is literally a question about physical ability (yes, anyone can physically pass salt). But pragmatically, it's a polite request. No human says "Yes, I can" and then doesn't pass it. But a naive NLP system that only does semantic analysis would make exactly that mistake.

Intent Detection Classify what action or goal a user has in mind when they write something. "What's the weather like in Chennai?" → intent: weather_query. "Book a table for two at 8pm" → intent: restaurant_reservation. "Cancel my subscription" → intent: cancellation. Intent detection is the core component of every chatbot and voice assistant — it determines which workflow to trigger. Trained as a classification task on labelled (utterance, intent) pairs.
Discourse Analysis Understanding meaning that spans multiple sentences — how sentences connect and build on each other. "The car was expensive. But John bought it anyway." — the "but" signals contrast, and the second sentence only makes sense in the context of the first. Discourse analysis tracks how ideas flow across a conversation or document: is this sentence adding to the previous point? Contradicting it? Providing evidence for it? Crucial for summarisation and dialogue management.
Speech Acts The theory that utterances perform actions. A promise (I will do X), a question (requesting information), a command (telling someone to do X), an assertion (claiming something is true), a declaration (pronouncing something into being — "You are fired"). John Austin and John Searle's speech act theory from linguistics has been applied to NLP for dialogue management: every turn in a conversation is one of these act types, and the system must track them to respond appropriately.
Context Understanding Using the surrounding conversational or document context to resolve ambiguities that can't be resolved from a single sentence. "Can I have it in blue?" only makes sense if the context established what "it" is. "What about tomorrow?" only makes sense relative to a previously mentioned date. In multi-turn dialogue systems, context is typically maintained in a dialogue state — a structured summary of what has been established so far. Modern LLMs handle this via their context window.
Pragmatics is what makes LLMs feel intelligent

When you ask Claude "Can you help me write an email to apologise to my boss for missing the deadline?", Claude doesn't just answer "yes" (semantic response to a yes/no question). It immediately understands the pragmatic intent — a request for help — and starts writing the email. This ability to interpret language at the pragmatic level, to understand intent and context, is the primary reason conversational LLMs feel so much more natural than earlier chatbots that only did intent classification from a fixed menu of options.

Part 8 · NLP Tasks You Can Build
From theory to production

Twelve NLP tasks — what you can actually build

The four levels of analysis aren't just academic — they directly power the NLP tasks that appear in production systems. Every task below maps to one or more of the analysis levels above.

Text Understanding
  • Text Classification — assign a label to a document. Spam/not spam, topic category, language detection. Built on Lexical + Semantic.
  • Sentiment Analysis — positive/negative/neutral or aspect-level. Review mining, brand monitoring. Semantic level.
  • Spam Detection — classify emails or messages as spam. Text classification specialisation.
  • Topic Modelling — unsupervised discovery of recurring themes across a corpus. LDA, NMF algorithms.
Text Generation
  • Machine Translation — translate between languages. English → French, Hindi → English. Requires all four levels.
  • Text Summarisation — extractive (select important sentences) or abstractive (rewrite in fewer words). Semantic + discourse.
  • Question Answering — given a document and a question, extract or generate the answer. RAG is QA at scale.
  • Chatbot / Dialogue — maintain coherent multi-turn conversation. Pragmatic analysis core.
Information Extraction
  • Named Entity Recognition — find and classify entities: people, organisations, dates, locations, monetary values.
  • Relation Extraction — extract structured relationships between entities. Builds knowledge graphs.
  • Event Detection — identify when specific events happen in text. "Apple launched the iPhone 16 on September 9."
  • Keyword Extraction — identify the most important terms in a document. TF-IDF, RAKE, or neural approaches.
Which tasks will you actually use when building agents

Intent Detection — routing user requests to the right agent or tool. Named Entity Recognition — extracting structured information from user messages and documents. Semantic Similarity — matching user queries to relevant documents in RAG. Sentiment Analysis — understanding customer feedback, escalation detection. Text Classification — labelling incoming data for routing, filtering, or prioritisation. Summarisation — compressing long context to fit in the agent's context window.

import nltk

nltk.download('stopwords', quiet=True)   # stop word lists
nltk.download('wordnet',   quiet=True)   # WordNet lexical database — needed for lemmatisation
nltk.download('averaged_perceptron_tagger', quiet=True)  # POS tagger model

import re                               # re: regular expressions — for pattern-based text cleaning
from nltk.tokenize import word_tokenize, sent_tokenize
# word_tokenize: split text into individual word/punctuation tokens
# sent_tokenize: split text into sentences
from nltk.corpus import stopwords
# stopwords: list of common English stop words ("the", "a", "is", "in", ...)
from nltk.stem import PorterStemmer, WordNetLemmatizer
# PorterStemmer: rule-based stemmer (crude but fast)
# WordNetLemmatizer: proper lemmatiser using WordNet dictionary

print("✅ Libraries ready")

# ── The full preprocessing pipeline ───────────────────────────
raw_text = "The CATS were Running quickly! Check https://example.com for more details. It's Amazing!!!"

# Step 1: Lowercase ─────────────────────────────────────────────
text = raw_text.lower()
# .lower(): converts every character to lowercase
# "CATS" → "cats", "Running" → "running", "Amazing" → "amazing"
print(f"1. Lowercased:  {text}")

# Step 2: Remove URLs and punctuation ───────────────────────────
text = re.sub(r'http\S+', '', text)
# re.sub(pattern, replacement, string): replace pattern matches with replacement
# r'http\S+': raw string, matches "http" followed by any non-whitespace characters (\S+)
# replaces URLs with empty string (removes them)
text = re.sub(r'[^a-z\s]', '', text)
# [^a-z\s]: any character that is NOT a lowercase letter or whitespace
# removes punctuation, numbers, special characters, apostrophes
text = text.strip()              # .strip(): remove leading and trailing whitespace
print(f"2. Cleaned:     {text}")

# Step 3: Tokenise ─────────────────────────────────────────────
tokens = word_tokenize(text)
# word_tokenize: uses NLTK's punkt tokeniser
# handles contractions and edge cases better than simple .split()
print(f"3. Tokenised:   {tokens}")

# Step 4: Remove stop words ─────────────────────────────────────
stop_words = set(stopwords.words('english'))
# set(): convert list to set for O(1) lookup — much faster than list membership check
filtered = [w for w in tokens if w not in stop_words]
# list comprehension: keep word only if it's NOT in the stop words set
print(f"4. No stopwords: {filtered}")

# Step 5: Stem vs Lemmatise ─────────────────────────────────────
stemmer    = PorterStemmer()
lemmatizer = WordNetLemmatizer()

stemmed    = [stemmer.stem(w)        for w in filtered]
lemmatized = [lemmatizer.lemmatize(w) for w in filtered]
# .stem(): rule-based cutting — may produce non-words
# .lemmatize(): proper dictionary form — always a real word

print(f"5a. Stemmed:     {stemmed}")
print(f"5b. Lemmatized:  {lemmatized}")
print(f"\nVocabulary size: {len(set(lemmatized))} unique tokens")

print(f"{'Token':10} {'Head':10} {'Relation':12} {'Meaning'"})
print("-" * 58)
for token in doc3:
    print(f"{token.text:10} {token.head.text:10} {token.dep_:12} {spacy.explain(token.dep_)}")
    # token.head: the word this token depends on (the governor)
    # token.dep_: the type of dependency relationship
    # nsubj = nominal subject, dobj = direct object, amod = adjectival modifier
    # det = determiner, prep = prepositional modifier, pobj = object of preposition

# ── Extract subject-verb-object triplets ─────────────────────
print("\n\n=== Subject-Verb-Object Extraction ===\n")
for token in doc3:
    if token.dep_ == "nsubj":        # find the subject of the sentence
        verb = token.head             # the verb is the subject's head
        subject = token
        obj = None
        for child in verb.children:  # children: all tokens that depend on this verb
            if child.dep_ == "dobj":  # dobj = direct object
                obj = child
        if obj:
            print(f"Subject: '{subject.text}' → Verb: '{verb.text}' → Object: '{obj.text}'")
            print(f"Triplet: ({subject.text}, {verb.text}, {obj.text})")
            # this triplet could go directly into a knowledge graph

# ── Zero-shot classification — classify without training data
classifier = pipeline(
    "zero-shot-classification",
    model="facebook/bart-large-mnli"
    # BART: Facebook's sequence-to-sequence model
    # fine-tuned on MNLI: Multi-Genre Natural Language Inference dataset
    # zero-shot: classify into ANY category you define — no training needed
)
print("✅ Zero-shot model loaded\n")

# ── Define user intents for an AI agent ──────────────────────
agent_intents = [
    "book_flight",
    "cancel_order",
    "check_weather",
    "customer_complaint",
    "technical_support",
    "general_question",
]
# these are the intents our agent handles — defined by us, no labelled data needed

# ── Test user messages ────────────────────────────────────────
user_messages = [
    "I need to fly to Mumbai next Friday, morning preferred",
    "My laptop keeps crashing when I open Chrome",
    "This is ridiculous, my order has been wrong three times now",
    "What's the temperature in Delhi today?",
]

print("=== Intent Detection for AI Agent ===\n")
for message in user_messages:
    result = classifier(
        message,
        candidate_labels=agent_intents,   # the intents to score against
        multi_label=False                 # False: pick one best intent (not multiple)
    )
    top_intent = result['labels'][0]      # labels sorted by score, highest first
    top_score  = result['scores'][0]
    second_intent = result['labels'][1]  # second best intent (for confidence check)
    second_score  = result['scores'][1]
    print(f"Message: '{message}'")
    print(f"  → Intent: {top_intent} ({top_score:.3f})")
    print(f"  → Runner-up: {second_intent} ({second_score:.3f})")
    print()

Part 10 · The Agentic AI Connection
Why this entire issue matters for AI Agents

NLP is the interface between humans and every agent you'll ever build

Every AI agent starts with a human instruction in natural language. Before any tool is called, any API is hit, any code is executed — that instruction must be understood. The preprocessing pipeline cleans the input. Lexical analysis identifies the entities. Semantic analysis extracts the intent and the parameters. Pragmatic analysis resolves what the user actually wants versus what they literally typed.

NER in agents: when a user says "Schedule a meeting with Sarah from finance for next Thursday at 2pm", an agent must extract the person (Sarah), the department (finance), the date (next Thursday = a specific date resolved from today), and the time (2pm). That's NER + temporal resolution — two NLP tasks running before a single calendar API is called.

Intent detection in agents: every multi-tool agent uses intent classification to decide which tool or workflow to trigger. Zero-shot classification (Cell 5) means you can define new intents at any time without retraining — just add a new label. This is why modern agent frameworks let you describe tools in plain English — the LLM does zero-shot classification over the tool descriptions to pick the right one.

Semantic similarity in agents: when an agent retrieves documents from a vector database (RAG), it computes semantic similarity between the user's query and stored document embeddings. That similarity measure — cosine distance between embedding vectors — is the practical implementation of everything the Semantic Analysis level is about. Same concept, different implementation. Block 04 Part 1 — the complete reference

Level 1 · Lexical Analysis
What are the words?
  • Tokenisation — split into words/sentences
  • POS Tagging — noun, verb, adjective labels
  • Morphological analysis — word structure
  • Named Entity Recognition — people, orgs, places
  • Library: spaCy · NLTK
Level 2 · Syntactic Analysis
How do words relate structurally?
  • Parsing — grammatical structure as a tree
  • Dependency parsing — subject/verb/object
  • Constituency parsing — nested phrase groups
  • CFG — formal grammar rule system
  • Library: spaCy · NLTK
Level 3 · Semantic Analysis
What does it mean?
  • Word sense disambiguation — bank = which bank?
  • Semantic similarity — same meaning, different words
  • Sentiment analysis — positive/negative/neutral
  • Coreference resolution — who is "he"?
  • Relation extraction → knowledge graphs
Level 4 · Pragmatic Analysis
What is the intent?
  • Intent detection — what does the user want?
  • Discourse analysis — meaning across sentences
  • Context understanding — resolve ambiguity
  • Speech acts — question, command, promise
  • Core of chatbots and dialogue systems
Preprocessing — always-on rules

Classical NLP → always preprocess (lowercase, clean noise, tokenise, remove stop words, lemmatise). Modern LLMs → skip preprocessing, feed raw text. Always lemmatise over stemming when output quality matters. When removing stop words: never for LLMs, always for TF-IDF/BoW. NER before any entity-dependent logic. Intent detection before any tool-calling logic.
