Part 1 · Why the Transformer
The paradigm shift

Why the Transformer replaced RNNs in 2017

The RNN/LSTM approach had three fundamental limits that no amount of engineering could fully fix:

RNN problemWhy it mattersTransformer's fix
Sequential processing Must process word 1, then word 2, then word 3 — in order. Can't be parallelised. Slow to train on GPUs designed for parallel computation. All tokens processed simultaneously. A sentence of 512 tokens doesn't take 512 steps — it takes 1 step with 512 parallel computations.
Vanishing gradients over distance Even with LSTM, connecting information 100+ tokens apart is hard. Gradients weaken across long paths. Attention directly connects any two tokens regardless of distance. "bank" and "river" 50 tokens apart have a direct, unattenuated connection.
Fixed-size bottleneck In Seq2Seq, the entire input must compress into one hidden state. Catastrophic for long documents. No bottleneck. Every output position can directly attend to every input position. All information preserved.

The Transformer doesn't process sequence by sequence. It processes the entire sequence at once — in one unified operation — and lets every position learn directly from every other.

Part 3 · Input Preparation
Step 1 of the block

Input Embedding + Positional Encoding

Before the Transformer can process tokens, two things must happen. First, each token must be converted into a dense vector — an embedding. Second, the model must somehow know the order of tokens, because unlike RNNs that process sequentially, the Transformer sees everything at once and has no inherent sense of position.

Token Embedding A lookup table (embedding matrix) of shape (vocabulary_size × d_model). Each token ID maps to one row — a learned vector of d_model dimensions. BERT uses d_model=768. GPT-3 uses d_model=12,288. The embedding matrix is one of the largest components of the model (50,000 tokens × 768 dims = 38.4M parameters for BERT alone) and is trained jointly with everything else via backprop.
Positional Encoding A vector added to each token embedding to inject position information. Without this, "cat sat on mat" and "mat on sat cat" would produce identical representations — same tokens, same attention, no order. The original 2017 Transformer used sinusoidal encoding: fixed mathematical functions of position and dimension. Modern models use learned positional embeddings (just another lookup table) or RoPE/ALiBi for better generalisation to long sequences.

The same word "cat" at position 2 vs position 7 produces different input embeddings — position is baked in. The model learns to use these position signals to understand word order.

Sinusoidal vs Learned Positional Encoding

Original Transformer (2017): sinusoidal — PE(pos, 2i) = sin(pos/10000^(2i/d_model)). Fixed, not trained. Can generalise to longer sequences than seen in training. BERT and GPT: learned — just another embedding table, trained via backprop like everything else. RoPE (Rotary Position Embedding, used in LLaMA and many modern models): encodes position by rotating Q and K vectors in the attention computation, enabling better length generalisation. ALiBi (used in Mistral): biases attention scores directly by distance. Position encoding is an active research area.

Part 4 · Multi-Head Self-Attention
The core of the Transformer

Multi-Head Self-Attention — every token attends to every other

You built single-head attention from scratch in Issue 07. Multi-Head Attention runs H attention operations in parallel — H "heads" — each with its own learned Q, K, V projection matrices. Each head learns to attend to different types of relationships simultaneously.

Why multiple heads?

Consider the sentence "The bank near the river where I fish is closed." A single attention head must choose what to focus on. But this sentence has multiple simultaneous relationships: syntactic (what modifies what), semantic (what refers to what), coreference ("where I fish" modifies "river bank"). Multiple heads can each specialise — head 1 might learn syntactic dependencies, head 2 might learn semantic similarity, head 3 might learn coreference. All captured simultaneously. BERT-base has 12 heads. GPT-3 has 96 heads per layer. Research shows different heads genuinely specialise in different linguistic phenomena.

Multi-Head Attention — BERT-base example: 12 heads in parallel
Head 1
syntactic subject-verb
Head 2
coreference "he/she/it"
Head 3

Head 4

Heads 5–12

All 12 head outputs concatenated → projected with W_O matrix → single output
Each head gets d_k = d_model / H dimensions. BERT: 768/12 = 64 dims per head. Concat restores full 768 dims.
Self-Attention vs Cross-Attention Self-attention: Q, K, V all come from the same sequence. Each token attends to all other tokens in the same sequence — how tokens in "The cat sat" relate to each other. Cross-attention: Q comes from one sequence (decoder), K and V come from another sequence (encoder output). Allows the decoder to look at the encoder's representation of the input. Used in the Encoder-Decoder (T5-style) Transformer. Pure decoder models (GPT, Claude) only use self-attention.
Causal Mask In decoder-only models (GPT, Claude), during training a token must not attend to future tokens — it should only see past context, otherwise the model trivially "cheats" by reading the answer. The causal mask sets all future positions to −∞ before softmax, making their attention weights effectively zero. After softmax, only positions 1 through t receive any weight when processing position t. This forces the model to predict each token using only its left context — which is exactly the autoregressive generation task it must perform at inference.

The full Multi-Head Attention computation

Multi-Head Attention — per head h
Q_h = X · W_Q_h (project input to query space for head h)
K_h = X · W_K_h (project input to key space for head h)
V_h = X · W_V_h (project input to value space for head h)
head_h = softmax(Q_h · K_h^T / √d_k) · V_h
MultiHead(X) = Concat(head_1, …, head_H) · W_O
Part 5 · Add & LayerNorm
The stabiliser

Add & LayerNorm — why deep networks don't collapse

After Multi-Head Attention and after the FFN, the same pattern repeats: Add the block's input back to its output (residual connection), then apply Layer Normalisation. These two operations together are what makes it possible to stack 96 or more Transformer blocks without the training exploding or vanishing.

Residual Connection (Skip Connection) Instead of output = Attention(x), compute output = x + Attention(x). Add the original input back to the transformed output. This is the same skip connection from ResNet in Issue 05. Gradients can flow directly backward through the addition, bypassing the attention and FFN layers entirely. Without residuals, a 96-layer Transformer would have vanishing gradients — layers near the input would receive near-zero gradient signal and learn nothing. Residuals solve this completely.
Layer Normalisation Normalise the activations across the feature dimension (not the batch dimension, unlike Batch Normalisation). For each token independently, compute mean and variance across its d_model values, then normalise and scale. This keeps activations in a stable numerical range as they pass through many layers. Without it, after 96 layers of attention + FFN operations, activations would drift to extreme values and gradient flow would break. Applied after every Add operation (or, in modern variants like Pre-LN, before every sublayer).
Pre-LN vs Post-LN — a practical detail

The original 2017 paper used Post-LN: LayerNorm after the residual add. Modern models (GPT-2 onwards, LLaMA, Claude) use Pre-LN: LayerNorm before each sublayer. Pre-LN trains more stably and doesn't require warm-up learning rate schedules. It's now the default. The formula becomes: output = x + Attention(LayerNorm(x)) instead of output = LayerNorm(x + Attention(x)). Small difference, big practical improvement for very deep models.

Part 6 · Feed-Forward Network
The second sublayer

Feed-Forward Network — where knowledge is stored

After Multi-Head Attention + Add&LayerNorm, each token's representation passes through a simple two-layer MLP — independently for each token. No interaction between tokens here — it's purely positional (each token processed in isolation).

FFN Sublayer FFN(x) = Linear_2(GELU(Linear_1(x))). Two linear layers with a GELU activation between them. The inner dimension is typically 4× d_model: if d_model=768, the FFN expands to 3072 then contracts back to 768. This 4× expansion is where most of the model's parameters live — and research suggests this is where factual knowledge is stored. The FFN acts like a key-value memory: the first layer matches patterns, the activation gates them, the second layer writes the result. Mechanistic interpretability research has shown specific facts ("Paris is the capital of France") are stored in specific FFN neurons.
Attention vs FFN — what each component does

Attention = token mixer. It lets tokens communicate with each other — aggregating context from across the sequence. FFN = position-wise transformation. It processes each token independently — applying the model's stored knowledge to that token's current representation. Think of it this way: attention decides "which information from elsewhere is relevant to this token right now." FFN decides "given this token's current representation, what do I know about it?" Together, one block performs one round of context-gathering + knowledge-application. Stack 96 rounds and you get something capable of writing poetry and debugging code.

Part 7 · Stacking Blocks
How depth creates capability

Stacking N blocks — abstraction through depth

Each block takes in a sequence of token representations and outputs a transformed sequence of the same shape. Block 1 output feeds directly into Block 2. Block 2 into Block 3. All the way to Block N. Each block refines the representation — asking "given what all previous blocks have computed, what additional context and transformation should I apply?"

Layer rangeWhat is typically learned (from interpretability research)
Early layers (1–4)Surface syntax — POS patterns, local co-occurrence, basic word relationships. Attention heads attend to nearby tokens.
Middle layers (5–8)Syntactic structure — subject/verb/object dependencies, phrase boundaries. Coreference begins to form.
Late layers (9–12+)Semantic and task-specific representations — entity types, sentiment, factual knowledge, reasoning patterns.
Why you can't train just a few layers

A 1-layer Transformer can attend to context but has almost no capacity to store knowledge or perform complex reasoning. A 3-layer Transformer can parse basic syntax but not disambiguate meaning. The capabilities that make LLMs useful — multi-step reasoning, world knowledge, instruction following — require the kind of compositional abstraction that only emerges through many layers of iterative attention + FFN processing. GPT-3 needed 96 layers and 175 billion parameters to achieve its emergent abilities. This is why you can't just make 1 big attention layer — depth is not optional.

Part 8 · Three Transformer Variants
Same block — three different configurations

BERT, GPT, T5 — three answers to the same architecture

The Transformer block is the same in all three. What differs is: how you stack the blocks (encoder only, decoder only, or both), what pre-training task you use, and what capabilities emerge from each choice.

Variant 1 · Encoder
BERT-style
Reads the full sentence in both directions. Every token sees every other token. Builds the deepest contextual understanding of input.
  • No causal mask — full bidirectional attention
  • Pre-training: Masked LM + NSP
  • Produces rich contextual representations
  • Text classification, NER, QA
  • Sentence embeddings (SBERT)
Models: BERT · RoBERTa · ELECTRA · DeBERTa
Variant 2 · Decoder
GPT-style
Generates text left-to-right. Each token only attends to previous tokens — the causal mask prevents peeking at the future.
  • Causal mask — autoregressive generation
  • Pre-training: next token prediction (CLM)
  • Scales to enormous parameter counts
  • Text generation, chatbots, code, reasoning
  • In-context learning (few-shot prompting)
Models: GPT-2/3/4 · Claude · LLaMA · Gemini · Mistral
Variant 3 · Encoder-Decoder
T5-style
Encoder reads and understands the full input. Decoder generates the output. Cross-attention lets decoder attend to encoder states.
  • Encoder: bidirectional (no causal mask)
  • Decoder: causal + cross-attention to encoder
  • Pre-training: denoising / span masking
  • Machine translation, summarisation, QA
  • Any sequence-to-sequence task
Models: T5 · BART · mT5 · Flan-T5
Variant 1 — BERT

Encoder — BERT reads both ways

BERT (Bidirectional Encoder Representations from Transformers, Google 2018) was a landmark model because it pre-trained a Transformer encoder bidirectionally for the first time at scale. The key insight: for understanding tasks, you want to see the full sentence before making any prediction. A model that reads left to right can't use "river" to disambiguate "bank" when "river" comes after "bank".

Masked Language Modelling (MLM) BERT's pre-training task. Randomly mask 15% of tokens in the input (replace with [MASK] token). Train the model to predict the original word for each masked position. Because the model sees the full sentence (including words after the masked position), it must learn genuine bidirectional context. Contrast with GPT's left-to-right prediction: "The [MASK] sat on the mat" — BERT uses "sat" and "mat" to predict [MASK]="cat". This bidirectionality is what makes BERT excellent at understanding tasks.
When to use a BERT-style encoder in production

Classification tasks: feed the [CLS] token representation to a linear classifier. NER: feed each token representation to a token classifier. Semantic search: fine-tune with Sentence-BERT contrastive loss. Embeddings for RAG: run BERT/SBERT on each document chunk, store the [CLS] vector. The strength of encoders is representation quality for discriminative tasks — they produce the best embeddings for retrieval and classification. Their limitation: they don't generate text natively (no autoregressive decoder).

Variant 2 — GPT / Claude

Decoder — autoregressive generation

The decoder-only architecture is what powers GPT, Claude, LLaMA, Gemini, and essentially every conversational LLM. The key constraint: during training, each token can only attend to tokens that came before it — the causal mask enforces this. This means the pre-training task is natural: predict the next token. And the generation task is the exact same operation: predict the next token given all previous tokens, repeat until done.

Autoregressive Generation Generate one token at a time, appending each to the sequence, then feeding the full extended sequence back in to generate the next. To generate "The cat sat": run the model on your prompt → get probability distribution over all tokens → sample "The" → run the model on [prompt + "The"] → sample "cat" → run on [prompt + "The cat"] → sample "sat" → and so on. Each new token is conditioned on all previous tokens. This is computationally expensive but produces coherent, context-aware text.
In-Context Learning An emergent ability of large decoder models: the ability to learn a new task from examples shown in the prompt, without any weight updates. "Translate English to French: cat → chat, dog → chien, house → ?" The model "learns" the translation pattern from the examples in its context window and applies it to the new word. This works because the model has seen enough examples during pre-training to recognise and generalise the pattern. Smaller models lack this capability — it's a scale-dependent emergent behaviour.
Why the decoder won — the generation-understanding unification

Initially, BERT encoders dominated NLP benchmarks — they were better at understanding tasks. But as decoder models scaled (GPT-3, then GPT-4, Claude), something unexpected happened: with enough scale, decoders became excellent at understanding tasks too — by framing them as generation. "Sentiment of 'I loved it': " → generate "positive". "Name the entities in this text: " → generate a list. As decoder models got bigger, the generation framing covered nearly every NLP task. The decoder won the architecture war through sheer scale and the universality of next-token prediction as a pre-training objective.

Variant 3 — T5

Encoder-Decoder — sequence in, sequence out

Google's T5 (Text-to-Text Transfer Transformer) frames every NLP task as a text-to-text problem: give the input as text, get the output as text. Translation: "translate English to French: The cat sat" → "Le chat était assis". Summarisation: "summarise: [long article]" → "[short summary]". Question answering: "question: What is the capital of France? context: France is a country whose capital is Paris." → "Paris".

Cross-Attention In the Encoder-Decoder architecture, decoder blocks have an extra attention sublayer: cross-attention. The Query comes from the decoder's current hidden state (what it's generated so far). The Keys and Values come from the encoder's output (its representation of the full input). This lets the decoder "look at" relevant parts of the encoder's understanding when generating each output token — the same principle as the attention mechanism in the original Seq2Seq+Attention work from NLP Issue 07, but now fully parameterised within the Transformer framework.

Cell 1 — Positional Encoding: sinusoidal + visualise
import torch
import torch.nn as nn
import numpy as np
import math
torch.manual_seed(42)

# ── Sinusoidal Positional Encoding ────────────────────────────
class SinusoidalPositionalEncoding(nn.Module):
    """Original 2017 paper positional encoding — no learned parameters."""
    def __init__(self, d_model: int, max_len: int = 512, dropout: float = 0.1):
        super().__init__()
        self.dropout = nn.Dropout(dropout)

        pe = torch.zeros(max_len, d_model)
        # pe: (max_len, d_model) — one positional vector per possible position

        position = torch.arange(0, max_len).unsqueeze(1).float()
        # position: (max_len, 1) — integer positions 0, 1, 2, …, max_len-1

        div_term = torch.exp(
            torch.arange(0, d_model, 2).float() * -(math.log(10000.0) / d_model)
        )
        # div_term: the 1/10000^(2i/d_model) scaling factor for each dimension pair
        # using exp(log(x)) = x for numerical stability

        pe[:, 0::2] = torch.sin(position * div_term)
        # even dimensions: sin(position × div_term)
        pe[:, 1::2] = torch.cos(position * div_term)
        # odd  dimensions: cos(position × div_term)
        # different frequencies at different dimensions create unique position fingerprints

        pe = pe.unsqueeze(0)
        # add batch dimension: (1, max_len, d_model) — broadcasts across batch
        self.register_buffer('pe', pe)
        # register_buffer: saves pe as a non-trainable part of the module state

    def forward(self, x):
        x = x + self.pe[:, :x.size(1), :]
        # add positional encoding to input embedding — same shape, element-wise
        # pe[:, :seq_len, :] slices to actual sequence length
        return self.dropout(x)

# ── Test: create embeddings and add positional encoding ───────
d_model = 16      # small d_model for demonstration (real: 768 or 1024)

# token_embeddings: random vectors simulating learned word embeddings

Cell 2 — Multi-Head Self-Attention from scratch
import torch.nn.functional as F

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model: int, num_heads: int, dropout: float = 0.1):
        super().__init__()
        assert d_model % num_heads == 0, "d_model must be divisible by num_heads"

        self.d_model   = d_model
        self.num_heads = num_heads
        self.d_k       = d_model // num_heads
        # d_k: dimension per head. BERT-base: 768//12 = 64 dims per head

        self.W_q = nn.Linear(d_model, d_model)  # one big matrix covers all H heads
        self.W_k = nn.Linear(d_model, d_model)  # equivalent to H separate W_q_h matrices
        self.W_v = nn.Linear(d_model, d_model)
        self.W_o = nn.Linear(d_model, d_model)  # output projection after concat
        self.dropout = nn.Dropout(dropout)

    def split_heads(self, x, batch_size):
        """Reshape (batch, seq, d_model) → (batch, heads, seq, d_k)"""
        x = x.view(batch_size, -1, self.num_heads, self.d_k)
        # view: reshape tensor without copying data
        # -1 means infer seq_len automatically
        return x.transpose(1, 2)
        # transpose dims 1 and 2: (batch, seq, heads, d_k) → (batch, heads, seq, d_k)
        # this puts heads first so each head's computation is batched together

    def forward(self, x, mask=None):
        batch_size = x.size(0)

        # ── Project to Q, K, V and split into H heads ────────────
        Q = self.split_heads(self.W_q(x), batch_size)
        K = self.split_heads(self.W_k(x), batch_size)
        V = self.split_heads(self.W_v(x), batch_size)
        # each now has shape: (batch, num_heads, seq_len, d_k)

        # ── Scaled dot-product attention (all heads in parallel) ──
        scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)
        # Q @ K^T: (batch, heads, seq, d_k) @ (batch, heads, d_k, seq) = (batch, heads, seq, seq)
        # each head produces a (seq_len × seq_len) attention score matrix

        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))
            # mask == 0: positions to block (future tokens in decoder)
            # fill with -inf: after softmax these become 0 (exp(-inf) = 0)

        attn_weights = self.dropout(F.softmax(scores, dim=-1))
        # softmax over last dim (key positions): each query position's weights sum to 1

        # ── Weighted sum of Values ────────────────────────────────
        context = torch.matmul(attn_weights, V)
        # (batch, heads, seq, seq) @ (batch, heads, seq, d_k) = (batch, heads, seq, d_k)

        # ── Concatenate heads and project ─────────────────────────
        context = context.transpose(1, 2).contiguous()
        # transpose back: (batch, seq, heads, d_k)
        # .contiguous(): ensure memory layout is contiguous for view()
        context = context.view(batch_size, -1, self.d_model)
        # merge heads: (batch, seq, heads × d_k) = (batch, seq, d_model)

        return self.W_o(context), attn_weights
        # W_o: final linear projection. shape: (batch, seq, d_model)
        # also return attention weights for inspection

# ── Test Multi-Head Attention ─────────────────────────────────

→ Diagonal slightly higher — tokens most attend to themselves initially (untrained)
→ After training, off-diagonal entries encode linguistic relationships
→ Weights sum to exactly 1.0 per row — attention is always a probability distribution
Cell 3 — One Full Transformer Block
class TransformerBlock(nn.Module):
    """One complete Transformer block — the repeating unit of all Transformers."""
    def __init__(self, d_model: int, num_heads: int, d_ff: int, dropout: float = 0.1):
        super().__init__()

        # ── Attention sublayer ────────────────────────────────────
        self.attention = MultiHeadAttention(d_model, num_heads, dropout)
        self.norm1     = nn.LayerNorm(d_model)
        # LayerNorm: normalises across d_model dimension for each token independently
        # has learnable scale (gamma) and shift (beta) parameters

        # ── Feed-Forward sublayer ─────────────────────────────────
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff),      # expand: 64 → 256 (4× expansion)
            nn.GELU(),                     # GELU: smooth non-linearity (replaces ReLU)
            nn.Dropout(dropout),
            nn.Linear(d_ff, d_model),      # contract back: 256 → 64
        )
        self.norm2   = nn.LayerNorm(d_model)
        self.dropout = nn.Dropout(dropout)

    def forward(self, x, mask=None):
        # ── Pre-LN variant (modern standard) ──────────────────────

        # Sublayer 1: Multi-Head Self-Attention + Residual
        attn_out, weights = self.attention(self.norm1(x), mask)
        # Pre-LN: apply LayerNorm BEFORE attention (not after)
        x = x + self.dropout(attn_out)
        # Residual: add original x back. Gradient can flow through the + directly.

        # Sublayer 2: Feed-Forward Network + Residual
        ffn_out = self.ffn(self.norm2(x))
        # Pre-LN: normalise before FFN
        x = x + self.dropout(ffn_out)
        # Second residual: add attention output to FFN output

        return x, weights

# ── Stack N blocks to create a Transformer encoder ────────────
class MiniTransformerEncoder(nn.Module):
    def __init__(self, d_model, num_heads, d_ff, num_layers, vocab_size, max_len=512):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, d_model)
        # Embedding: lookup table, shape (vocab_size, d_model)
        self.pos_enc   = SinusoidalPositionalEncoding(d_model, max_len, dropout=0.1)
        self.blocks    = nn.ModuleList(
            [TransformerBlock(d_model, num_heads, d_ff) for _ in range(num_layers)]
        )
        # ModuleList: registers each block as a submodule (important for .parameters())
        self.norm_final = nn.LayerNorm(d_model)
        # final normalisation after all blocks

    def forward(self, token_ids, mask=None):
        x = self.embedding(token_ids)   # (batch, seq) → (batch, seq, d_model)
        x = self.pos_enc(x)             # add positional encoding
        for block in self.blocks:        # pass through each block in sequence
            x, _ = block(x, mask)
        return self.norm_final(x)        # final normalisation

Cell 4 — Causal Mask: the decoder's constraint
# ── Build the causal mask for decoder-only (GPT-style) models ─
def make_causal_mask(seq_len: int) -> torch.Tensor:
    """Create a causal (lower-triangular) attention mask.
    Returns a (seq_len, seq_len) boolean tensor.
    True  = this position CAN be attended to.
    False = this position is BLOCKED (future token).
    """
    mask = torch.tril(torch.ones(seq_len, seq_len))
    # torch.tril: lower triangle including diagonal, zeros above diagonal
    # tril[i][j] = 1 if j <= i (can attend to current or past)
    # tril[i][j] = 0 if j >  i (cannot attend to future)
    return mask.bool()

print("1 = CAN attend  |  0 = BLOCKED (future)\n")

Part 10 · Connection to AI Agents
The Transformer inside every agent you'll build

You just learned the internals of every LLM you'll ever call

When your agent calls anthropic.messages.create() or openai.chat.completions.create(), the model on the other side is this architecture — the Transformer block you just built, stacked 96 times (GPT-3) or more, trained on trillions of tokens. Understanding it changes how you reason about what the model can and can't do.

The causal mask (Cell 4) is why LLMs generate left-to-right and why you can't ask a decoder model to "fill in the blank in the middle" — it physically cannot attend to tokens after the blank during generation. The context window is the maximum sequence length the model handles (2048 tokens in GPT-2, 200K tokens in Claude 3.5). When you exceed it, the model literally cannot attend to the oldest tokens — they're outside the attention range.

Multi-head attention heads (different heads learning different relationship types) explain why LLMs are so good at code — one head learns syntax (brackets, indentation), another learns semantic scoping (which variable is in scope), another learns API patterns (what methods follow this object type). These specialisations emerged from training, not from being explicitly programmed.

The FFN sublayer is where factual knowledge is stored — this is why fine-tuning on domain-specific data (medical texts, legal documents) improves factual accuracy. You're loading new patterns into the FFN's key-value memory. RAG is an architectural alternative: instead of baking knowledge into the FFN via training, you retrieve it at inference time and inject it into the attention context. Both approaches exploit the same Transformer architecture — just at different stages.

The three variants define your architecture choice when building AI systems. BERT-style encoders produce the best embeddings for retrieval — use them as the backbone of your RAG vector store. GPT-style decoders power chat, generation, and instruction-following — use them as the brain of your agent. T5-style encoder-decoders are best for structured transformation tasks — summarisation pipelines, translation systems, document structuring.

Block 05 in one paragraph

The Transformer replaced RNNs because it processes all tokens in parallel and connects any two tokens directly. One block: token embeddings + positional encoding → multi-head self-attention (H heads, Q·K^T/√d → softmax → ×V) → Add & LayerNorm (residual) → FFN (expand 4× → GELU → contract) → Add & LayerNorm. Repeat N times. Three variants: Encoder (BERT — bidirectional, best for understanding + embeddings), Decoder (GPT/Claude — causal mask, autoregressive generation), Encoder-Decoder (T5 — cross-attention, sequence-to-sequence tasks).
