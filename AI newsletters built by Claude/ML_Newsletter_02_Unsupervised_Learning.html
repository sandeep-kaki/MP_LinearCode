From the editor

Welcome back to Learn ML From Me. Last issue we covered Supervised Learning — the machine that learns from your labels. This issue is about something fundamentally different.

Unsupervised learning is the part of ML that most beginners skip too fast. That's a mistake. Because some of the most powerful things AI does — finding customer segments you didn't know existed, compressing 1000-dimensional data into 2 dimensions, discovering that "people who buy X also buy Y" — none of that has a label. None of that is supervised.

And when you get to building AI Agents, unsupervised learning is everywhere: embeddings, semantic search, RAG, anomaly detection — all of it traces back to what we're covering today.

Let's get into it.

Part 1 · The Mental Model Shift
The one thing to understand first

Supervised vs Unsupervised — a completely different question

In supervised learning, you asked: "Given this input, what is the correct output?" You had labels. The model learned the mapping between input and label.

Unsupervised learning asks a completely different question:

"I have no labels, no correct answers, no one telling me what to look for. What structure, patterns, or groupings naturally exist in this data?"

The model doesn't know what "correct" looks like. Instead, it discovers — on its own — that certain data points are similar to each other, that certain dimensions are redundant, that certain items tend to appear together.

The analogy

Imagine you're a new librarian. You inherit a room with 10,000 unlabelled books dumped in a pile. Nobody tells you how to categorise them. Supervised learning is impossible — there are no labels. So you start reading, noticing patterns: these 400 books use similar vocabulary, these 200 share themes, these 50 keep referencing each other. You create categories from the data itself. That's unsupervised learning. You didn't classify books into predefined categories — you discovered the categories.

Unsupervised learning has three major flavours. Clustering — find groups of similar things. Dimensionality Reduction — compress data while keeping its essence. Association Rule Mining — find "if this, then that" patterns. Each solves a fundamentally different problem. Let's go through all of them.

Part 2 · Clustering
Clustering — Algorithm 1

K-Means — the simplest grouping algorithm that works

K-Means is the most widely used clustering algorithm. The idea is almost embarrassingly simple. You tell it: "I think there are K groups in this data." It finds them.

How it works — step by step

  • 01
    Initialise: Randomly place K points (called centroids) somewhere in your data space. These will become the centres of your K clusters.
  • 02
    Assign: For every data point, measure the distance to every centroid. Assign each point to its nearest centroid. Every point now belongs to a cluster.
  • 03
    Update: Recalculate each centroid as the mean (average position) of all points currently assigned to it. The centroid moves to the centre of its cluster.
  • 04
    Repeat: Go back to step 2 and reassign. Repeat steps 2–3 until the centroids stop moving (convergence). That's your final clustering.
Real world analogy

You run an e-commerce site and have 1 million customers with no labels. K-Means with K=4 might discover: Cluster 1 = high-value frequent buyers, Cluster 2 = price-sensitive deal hunters, Cluster 3 = occasional browsers, Cluster 4 = one-time purchasers. You never told it to find these segments. It found them purely from purchase frequency, average order value, and recency data.

The K problem — how do you choose K?

This is the central challenge of K-Means. You need to tell it how many clusters to find. If you don't know (and usually you don't), use the Elbow Method:

  • →
    Run K-Means for K = 1, 2, 3, 4, ... 10
  • →
    For each K, record the inertia (sum of distances from each point to its centroid — lower = tighter clusters)
  • →
    Plot K vs inertia. The curve will look like an arm with an elbow. The elbow point = the optimal K. Adding more clusters beyond the elbow gives diminishing returns.
K-Means limitations you must know

It assumes clusters are roughly spherical and similar in size — bad for irregular shapes. It's sensitive to outliers (one extreme point can pull a centroid far away). It can converge to a local minimum depending on random initialisation. Fix: run it multiple times (n_init=10) and pick the best result. Always scale features before K-Means — it uses distances, so unscaled features will dominate unfairly.

Clustering — Algorithm 2

Hierarchical Clustering — building a tree of groups

K-Means requires you to decide K upfront. What if you don't know how many clusters you want? Hierarchical clustering builds a full tree of clusterings from 1 group all the way down to N individual points — and you decide the cut-off after seeing the full picture.

How Agglomerative (bottom-up) works

  • 01
    Start: every single data point is its own cluster. N data points = N clusters.
  • 02
    Find the two closest clusters (by your chosen linkage: single, complete, or average linkage). Merge them into one cluster.
  • 03
    You now have N-1 clusters. Repeat — find the next two closest and merge.
  • 04
    Keep merging until everything is one big cluster. Record the full history of merges.
  • 05
    Visualise as a dendrogram — a tree diagram showing every merge and at what distance it happened. Cut the tree at any horizontal line to get any number of clusters you want.
K-Means
  • Need to specify K upfront
  • Fast on large datasets
  • Spherical clusters only
  • Good for millions of points
  • Can't see cluster hierarchy
Hierarchical
  • No need to specify K upfront
  • Slow on large datasets (O(n³))
  • Any shape of clusters
  • Best for hundreds/thousands
  • Full hierarchy visible in dendrogram
When to use Hierarchical over K-Means

When you genuinely don't know how many clusters exist and want to explore the data. When you need to understand cluster relationships (are these two groups subcategories of a bigger group?). When your dataset is small-to-medium (under ~10,000 points). Gene expression analysis and document taxonomy are classic use cases.

Part 3 · Association Rules
Association Rule Mining

Apriori — "people who bought X also bought Y"

This is an entirely different type of unsupervised learning. It's not about grouping data points — it's about finding co-occurrence patterns in transactions. Which items appear together? Which events co-occur?

The classic example

A supermarket has 10,000 transaction receipts. No labels. Apriori discovers: "87% of customers who buy bread and butter also buy milk in the same transaction." This is an association rule: {bread, butter} → {milk}. The supermarket puts milk near bread. Sales go up. Nobody labelled anything — the pattern came from the raw transactions.

The three key metrics

MetricWhat it meansExample
Support How often does this combination appear across all transactions? Filters out rare patterns. {bread, milk} appears in 40% of all transactions → support = 0.40
Confidence Given someone bought X, what % also bought Y? Measures the strength of the rule. Of customers buying bread, 75% also bought milk → confidence = 0.75
Lift Is this rule better than random chance? Lift > 1 means the association is real, not a coincidence. Lift = 2.5 means milk is 2.5x more likely to be bought when bread is in the cart vs by itself

How Apriori works

The algorithm's key insight is the Apriori property: if an itemset is rare (low support), then any larger itemset containing it is also rare. This lets it prune the search space massively instead of checking every possible combination.

  • 01
    Find all individual items that meet minimum support threshold (e.g., appear in at least 5% of transactions).
  • 02
    Use those frequent single items to generate candidate pairs. Check their support. Keep only frequent pairs.
  • 03
    From frequent pairs, generate candidate triples. Check support. Keep frequent triples.
  • 04
    Continue until no more frequent itemsets can be found.
  • 05
    Generate association rules from all frequent itemsets. Filter by confidence and lift thresholds.
Where Apriori shows up in AI systems

In recommendation engines (Netflix, Amazon, Spotify), knowledge graph construction, and medical diagnosis systems ("patients with symptoms A and B also present symptom C in 78% of cases"). When an AI agent is deciding what tool to call next based on conversation context — the pattern mining concept is the same idea.

Part 4 · Dimensionality Reduction
Dimensionality Reduction — Algorithm 1

PCA — compressing data without losing what matters

PCA (Principal Component Analysis) is one of the most important algorithms in all of machine learning. Not because it's complex — but because the problem it solves appears everywhere.

The problem: high-dimensional data. Imagine you have 1000 features describing each customer. Most of those features are correlated (income and spending are related; age and tenure are related). You have a lot of redundant information packed into 1000 dimensions. That's slow to compute, hard to visualise, and often causes overfitting.

PCA asks: "What is the minimum number of new dimensions I need to preserve most of the information in the data?"

The intuition — shadows on a wall

The shadow analogy

Hold a 3D object in front of a light. You see a 2D shadow on the wall. The shadow is a projection — it loses some information (depth), but if you choose the right angle to project from, the shadow captures the most important shape. PCA finds the mathematically optimal "angles" (called principal components) to project your data onto fewer dimensions while losing as little information as possible.

What PCA actually does — step by step

  • 01
    Standardise: Scale all features to mean=0, std=1. PCA is sensitive to scale — always do this first.
  • 02
    Compute covariance matrix: Find how every feature varies with every other feature. This tells PCA where the spread of the data is.
  • 03
    Find eigenvectors and eigenvalues: The eigenvectors are the directions of maximum spread (principal components). The eigenvalues tell you how much variance each direction captures.
  • 04
    Rank by explained variance: Sort principal components by eigenvalue (highest first). The first PC captures the most variance, second captures the second most, and so on.
  • 05
    Project: Choose how many components to keep (enough to explain 90–95% of variance). Project your data onto those components. Done — you've compressed N dimensions to K dimensions.
Explained Variance Ratio = eigenvalue_i / sum(all eigenvalues)
How many components to keep?

Plot the cumulative explained variance ratio as you add components. When the curve flattens past 90% or 95%, that's your cut-off. If 10 components out of 1000 explain 95% of variance — use 10. You've reduced the problem 100x with minimal information loss.

What PCA is NOT

Common misconception

PCA doesn't select which original features are important and discard the rest. It creates entirely new features (principal components) that are linear combinations of all original features. You lose interpretability — PC1 isn't "age" or "income", it's 0.3×age + 0.7×income - 0.1×tenure + ... The tradeoff: you gain compression and often better model performance.

Dimensionality Reduction — Algorithm 2

Autoencoders — PCA but with neural networks

PCA is linear — it can only find straight-line combinations of features. Real data is often non-linear. An autoencoder is a neural network that does the same job as PCA but can capture non-linear patterns.

The architecture — a bottleneck by design

The compression analogy

Imagine compressing a photo to send through a narrow pipe. You squeeze it down at one end (encoder), pass the tiny compressed version through the pipe (bottleneck — the latent space), and reconstruct it at the other end (decoder). The network is forced to learn only the most essential representation of the image to survive the bottleneck. If the reconstruction is good, the bottleneck captured the essence.

Encoder half
  • Takes full input (e.g. 784 pixels)
  • Compresses through shrinking layers
  • Outputs a small latent vector (e.g. 32 numbers)
  • This is the compressed representation
Decoder half
  • Takes the small latent vector
  • Expands through growing layers
  • Tries to reconstruct the original input
  • Loss = reconstruction error

The key insight: we never use the decoder after training. We throw it away. The encoder is what we want — it's a function that compresses any input into a meaningful low-dimensional representation. That representation is called the latent space.

PCA vs Autoencoder — when to use which

PCA: Fast, interpretable, always works, linear relationships only. Use for tabular data, quick dimensionality reduction, preprocessing before another model. Autoencoder: Slower to train, handles non-linear patterns, needs more data. Use for images, audio, text, anything with complex non-linear structure. The embeddings that power semantic search in RAG systems? Those are autoencoder-style neural network compressions.

Autoencoders in production AI

The embedding models you'll use in LLM applications (like text-embedding-3-small from OpenAI) are essentially sophisticated autoencoders trained on massive text corpora. They compress any piece of text into a 1536-number vector that captures its semantic meaning. That's the same bottleneck architecture — just at massive scale, with transformer blocks instead of simple linear layers.

Part 5 · When to Use Which
Decision guide

Picking the right unsupervised algorithm

Your goalAlgorithmCondition
Group customers / users / documents K-Means Large dataset, roughly equal-sized groups, speed matters
Group data but don't know K Hierarchical Small-medium dataset, want to explore full hierarchy
Find "also bought" / co-occurrence patterns Apriori Transaction or event data, looking for if-then rules
Speed up another ML model, reduce noise PCA Tabular data, many correlated features, need interpretability
Compress images, text, or complex non-linear data Autoencoder Enough data to train a neural network, non-linear patterns
Visualise high-dimensional data in 2D PCA (first 2 components) Quick exploration of any dataset — always try this first
Detect anomalies (fraud, defects) Autoencoder Normal data compresses well; anomalies have high reconstruction error

Cell 1 — Setup & Imports
# ── Import everything we need ──────────────────────────────────
import numpy as np                      # fast math on arrays
import pandas as pd                    # DataFrames for tabular data
import matplotlib.pyplot as plt        # plotting
import seaborn as sns                   # prettier statistical plots
import warnings
warnings.filterwarnings('ignore')       # silence non-critical warnings

# ── Clustering algorithms ──────────────────────────────────────
from sklearn.cluster import KMeans, AgglomerativeClustering
# KMeans: partition into K groups by distance
# AgglomerativeClustering: bottom-up hierarchical clustering

# ── Dimensionality reduction ───────────────────────────────────
from sklearn.decomposition import PCA
# PCA: Principal Component Analysis — linear compression

# ── Data tools ────────────────────────────────────────────────
from sklearn.preprocessing import StandardScaler
# StandardScaler: mean=0, std=1 — mandatory before K-Means and PCA
from sklearn.datasets import make_blobs, load_iris
# make_blobs: generates realistic clustered data for demos
# load_iris: classic real-world dataset (150 flowers, 4 features)
from sklearn.metrics import silhouette_score
# silhouette_score: measures clustering quality without needing labels
# ranges from -1 (bad) to +1 (perfect), 0.5+ is generally good

# ── Dendrogram (for hierarchical clustering) ──────────────────
from scipy.cluster.hierarchy import dendrogram, linkage
# scipy has a better dendrogram tool than sklearn
# linkage: computes the merge history for hierarchical clustering
# dendrogram: visualises that merge history as a tree

np.random.seed(42)         # reproducibility — same random results every run
print("✅ Libraries ready")
Cell 2 — K-Means + Elbow Method
# ── Generate realistic clustered data ─────────────────────────
X_blobs, true_labels = make_blobs(
    n_samples=300,         # 300 data points total
    centers=4,              # 4 true underlying clusters
    cluster_std=0.8,        # how spread out each cluster is (lower = tighter)
    random_state=42         # reproducible
)
# true_labels exist here only for visual comparison — K-Means never sees them

# ── Scale features BEFORE K-Means ─────────────────────────────
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X_blobs)
# always scale first — K-Means uses distances, unscaled features dominate

# ── Elbow Method — find the best K ────────────────────────────
inertia_values = []                     # will store the "tightness" score for each K
K_range = range(1, 11)                  # try K = 1, 2, 3, ..., 10

for k in K_range:
    km = KMeans(
        n_clusters=k,         # number of clusters to find
        n_init=10,            # run K-Means 10 times with different random starts
        random_state=42       # reproducible
    )
    km.fit(X_scaled)          # find K clusters in the data
    inertia_values.append(km.inertia_)
    # km.inertia_ = sum of squared distances from each point to its centroid
    # lower inertia = tighter, more compact clusters

# ── Plot the elbow curve ───────────────────────────────────────
fig, axes = plt.subplots(1, 2, figsize=(14, 5))
# subplots(1, 2) = one row, two charts side by side

axes[0].plot(K_range, inertia_values,
             marker='o',          # put a circle dot at each K value
             color='#7c3aed',     # purple line
             linewidth=2.5,
             markersize=7)
axes[0].axvline(x=4,               # vertical line at the elbow (K=4)
               color='tomato',
               linestyle='--',     # dashed line
               label='Elbow at K=4')
axes[0].set_xlabel('Number of Clusters (K)')
axes[0].set_ylabel('Inertia (lower = tighter clusters)')
axes[0].set_title('Elbow Method — Finding Optimal K')
axes[0].legend()
axes[0].grid(True, alpha=0.3)

# ── Train final K-Means with optimal K=4 ──────────────────────
final_km = KMeans(n_clusters=4, n_init=10, random_state=42)
final_km.fit(X_scaled)               # train on scaled data
cluster_labels = final_km.labels_     # predicted cluster for each data point
centroids = final_km.cluster_centers_ # coordinates of the 4 cluster centres

# ── Plot final clusters ────────────────────────────────────────
colors = ['#7c3aed', '#0d7377', '#f59e0b', '#e11d48']
# one colour per cluster

for i in range(4):
    mask = cluster_labels == i         # boolean mask: True for points in cluster i
    axes[1].scatter(
        X_scaled[mask, 0],            # x-coordinate of all points in cluster i
        X_scaled[mask, 1],            # y-coordinate of all points in cluster i
        c=colors[i], s=45, alpha=0.8,
        label=f'Cluster {i+1}'
    )
axes[1].scatter(
    centroids[:, 0], centroids[:, 1],  # x and y of all 4 centroids
    c='black', marker='X',              # X marker = centroid
    s=200, zorder=5,                    # zorder=5 draws on top of everything
    label='Centroids'
)
axes[1].set_title('K-Means Result — K=4')
axes[1].legend()
axes[1].grid(True, alpha=0.2)
plt.tight_layout()
plt.show()

# ── Evaluate without labels using silhouette score ─────────────
sil = silhouette_score(X_scaled, cluster_labels)
print(f"Silhouette Score: {sil:.3f}  (range: -1 to +1,  0.5+ = good clustering)")
Cell 3 — Hierarchical Clustering + Dendrogram
# ── Use a smaller subset for clarity in the dendrogram ────────
X_small = X_scaled[:50]
# take just 50 points — dendrogram gets messy with too many points

# ── Compute linkage matrix (full merge history) ────────────────
Z = linkage(
    X_small,
    method='ward'    # ward = minimise variance within clusters when merging
                      # other options: 'complete' (max distance), 'average'
)
# Z is a matrix where each row = one merge: [cluster_a, cluster_b, distance, count]

# ── Plot the dendrogram ────────────────────────────────────────
fig, axes = plt.subplots(1, 2, figsize=(15, 5))

dendrogram(
    Z,
    ax=axes[0],                       # which subplot to draw on
    truncate_mode='lastp',            # only show the last p merges (cleaner)
    p=12,                              # show last 12 merges
    leaf_rotation=45,                 # rotate x-axis labels 45 degrees
    color_threshold=3.5               # colour clusters below this distance
)
axes[0].set_title('Dendrogram — Hierarchical Clustering')
axes[0].set_xlabel('Data points (or cluster size)')
axes[0].set_ylabel('Distance at merge (cut here to choose K)')
axes[0].axhline(y=3.5, color='tomato', linestyle='--',
                 label='Cut line → 4 clusters')
# axhline draws a horizontal line — where you cut = how many clusters you get
axes[0].legend()

# ── Apply Agglomerative Clustering with K=4 ───────────────────
agg = AgglomerativeClustering(
    n_clusters=4,       # cut the tree at 4 clusters
    linkage='ward'       # same linkage as above for consistency
)
agg_labels = agg.fit_predict(X_small)
# fit_predict = fit and return labels in one step (no separate predict needed)

# ── Plot hierarchical clustering result ───────────────────────
for i in range(4):
    mask = agg_labels == i
    axes[1].scatter(
        X_small[mask, 0],
        X_small[mask, 1],
        c=colors[i], s=60, alpha=0.85,
        label=f'Cluster {i+1}',
        edgecolors='white',           # white border around each dot
        linewidths=0.5
    )
axes[1].set_title('Agglomerative Clustering — K=4')
axes[1].legend()
axes[1].grid(True, alpha=0.2)
plt.tight_layout()
plt.show()
Cell 4 — Apriori / Association Rules
# ── Install mlxtend (has Apriori implementation) ──────────────
# Run this once. After install, skip this cell on re-runs.
import subprocess
subprocess.run(['pip', 'install', 'mlxtend', '-q'])
# subprocess.run: runs a terminal command from inside Python
# -q = quiet mode (less output noise)

from mlxtend.frequent_patterns import apriori, association_rules
# apriori: finds all itemsets that meet minimum support
# association_rules: generates if-then rules from those itemsets
from mlxtend.preprocessing import TransactionEncoder
# TransactionEncoder: converts transaction lists to a binary matrix

# ── Create a fake grocery transaction dataset ──────────────────
transactions = [
    ['bread', 'milk', 'butter'],            # customer 1 bought these
    ['bread', 'milk'],                       # customer 2
    ['milk', 'eggs', 'cheese'],             # customer 3
    ['bread', 'butter', 'eggs'],             # customer 4
    ['bread', 'milk', 'eggs', 'butter'],    # customer 5
    ['milk', 'butter'],                      # customer 6
    ['bread', 'eggs'],                       # customer 7
    ['bread', 'milk', 'cheese'],             # customer 8
    ['eggs', 'butter', 'cheese'],            # customer 9
    ['bread', 'milk', 'butter', 'eggs'],    # customer 10
]

# ── Encode: transaction lists → binary True/False matrix ──────
te = TransactionEncoder()
te_array = te.fit_transform(transactions)
# each row = one transaction, each column = one item
# True = item was in that transaction, False = was not

df_trans = pd.DataFrame(te_array, columns=te.columns_)
# wrap in a DataFrame so apriori can read it
print("Transaction matrix (True = item was bought):\n")
print(df_trans.to_string())   # to_string() shows all rows without truncation

# ── Run Apriori to find frequent itemsets ─────────────────────
frequent_itemsets = apriori(
    df_trans,
    min_support=0.4,          # only keep itemsets appearing in ≥40% of transactions
    use_colnames=True         # show item names instead of column numbers
)
print(f"\nFrequent itemsets (support ≥ 40%):\n")
print(frequent_itemsets.sort_values('support', ascending=False).to_string())
# sort_values: sort DataFrame rows by the 'support' column, highest first

# ── Generate association rules from frequent itemsets ──────────
rules = association_rules(
    frequent_itemsets,
    metric='confidence',       # filter rules by confidence
    min_threshold=0.6          # keep only rules with confidence ≥ 60%
)
print(f"\nAssociation Rules (confidence ≥ 60%):\n")
print(rules[['antecedents', 'consequents', 'support', 'confidence', 'lift']]
      .sort_values('lift', ascending=False)
      .round(3)
      .to_string())
# antecedents = the "if" part,  consequents = the "then" part
# round(3) = show 3 decimal places
# sort by lift: highest lift = strongest real association (not just coincidence)
Cell 5 — PCA on the Iris Dataset
# ── Load the Iris dataset — 150 flowers, 4 features ───────────
iris = load_iris()              # loads the dataset object
X_iris = iris.data               # features: sepal length/width, petal length/width
y_iris = iris.target             # labels: 0=Setosa, 1=Versicolor, 2=Virginica
species = iris.target_names      # array: ['setosa', 'versicolor', 'virginica']

print(f"Original shape: {X_iris.shape}")
# shape = (150, 4) → 150 flowers, 4 features each

# ── Scale before PCA — mandatory ──────────────────────────────
sc_iris = StandardScaler()
X_iris_scaled = sc_iris.fit_transform(X_iris)
# PCA finds directions of maximum variance
# if features aren't scaled, the high-variance feature dominates unfairly

# ── Run PCA — keep ALL components first to study explained variance
pca_full = PCA()               # no n_components argument = keep all
pca_full.fit(X_iris_scaled)     # learn the principal component directions

# explained_variance_ratio_: % of variance captured by each component
evr = pca_full.explained_variance_ratio_
cumulative_evr = np.cumsum(evr)
# np.cumsum: running total — [0.73, 0.73+0.23, 0.73+0.23+0.03, ...]

fig, axes = plt.subplots(1, 3, figsize=(16, 5))

# ── Plot 1: explained variance per component ──────────────────
axes[0].bar(
    range(1, 5),              # x-axis: component numbers 1 to 4
    evr * 100,                # y-axis: % explained (multiply by 100 for percentage)
    color='#7c3aed', alpha=0.8
)
axes[0].plot(
    range(1, 5),
    cumulative_evr * 100,     # plot cumulative % on top
    marker='o', color='tomato', linewidth=2
)
axes[0].axhline(y=95, color='green', linestyle='--', label='95% threshold')
axes[0].set_xlabel('Principal Component')
axes[0].set_ylabel('Explained Variance (%)')
axes[0].set_title('How much does each PC explain?')
axes[0].legend()
axes[0].grid(True, alpha=0.3)

# ── Reduce to 2 components (for 2D visualisation) ─────────────
pca_2d = PCA(n_components=2)           # keep only 2 principal components
X_pca = pca_2d.fit_transform(X_iris_scaled)
# X_pca shape: (150, 2) — compressed from 4 features to 2
print(f"Compressed shape: {X_pca.shape}")
print(f"Variance retained: {pca_2d.explained_variance_ratio_.sum()*100:.1f}%")

# ── Plot 2: raw data (can't visualise 4D) vs PCA 2D ───────────
axes[1].set_title('Original data — 4D (can\'t plot)')
axes[1].text(0.5, 0.5, '4 dimensions\n← can\'t visualise →',
             ha='center', va='center',
             fontsize=14, color='gray',
             transform=axes[1].transAxes)
# transAxes: position text using 0-1 coordinates relative to axes (not data)
axes[1].axis('off')            # hide axis lines for this placeholder

# ── Plot 3: PCA 2D — now we CAN visualise and see the clusters
palette = ['#7c3aed', '#0d7377', '#f59e0b']   # one colour per species
for i, sp_name in enumerate(species):
    # enumerate: gives (index, value) pairs → (0, 'setosa'), (1, 'versicolor')...
    mask = y_iris == i          # True for all flowers of species i
    axes[2].scatter(
        X_pca[mask, 0],         # PC1 values for this species
        X_pca[mask, 1],         # PC2 values for this species
        c=palette[i], s=55, alpha=0.85,
        edgecolors='white', linewidths=0.5,
        label=sp_name

Cell 6 — Autoencoder with PyTorch
import torch                           # PyTorch: deep learning framework
import torch.nn as nn                  # nn: neural network building blocks
import torch.optim as optim            # optim: optimisers (Adam etc)
from torch.utils.data import DataLoader, TensorDataset
# DataLoader: feeds batches of data to the model during training
# TensorDataset: wraps tensors into a dataset PyTorch can iterate

# ── Convert Iris data to PyTorch tensors ──────────────────────
X_tensor = torch.FloatTensor(X_iris_scaled)
# FloatTensor: PyTorch works with tensors (like numpy arrays but GPU-compatible)
# FloatTensor = 32-bit floating point numbers (standard for neural networks)

dataset = TensorDataset(X_tensor, X_tensor)
# NOTICE: X_tensor appears twice — input AND target are the same!
# Autoencoder tries to reconstruct its own input — that's the whole point

loader = DataLoader(dataset, batch_size=32, shuffle=True)
# batch_size=32: feed 32 examples at a time during training
# shuffle=True: randomise order each epoch (prevents the model learning order)

# ── Define the Autoencoder architecture ───────────────────────
class Autoencoder(nn.Module):          # inherit from nn.Module — every PyTorch model does this
    def __init__(self):
        super().__init__()              # initialise parent class (required)

        # ── Encoder: 4 features → 2 (the bottleneck) ──────────────
        self.encoder = nn.Sequential(
            # Sequential: run layers one after another in order
            nn.Linear(4, 8),            # fully connected: 4 inputs → 8 hidden neurons
            nn.ReLU(),                  # ReLU activation: max(0, x) — adds non-linearity
            nn.Linear(8, 2)             # 8 hidden → 2 (our bottleneck = 2D latent space)
        )

        # ── Decoder: 2 (bottleneck) → 4 features (reconstruct) ───
        self.decoder = nn.Sequential(
            nn.Linear(2, 8),            # 2 latent → 8 hidden
            nn.ReLU(),                  # add non-linearity
            nn.Linear(8, 4)             # 8 hidden → 4 outputs (same as original input)
        )

    def forward(self, x):               # forward: defines what happens on each pass
        latent = self.encoder(x)       # compress: 4D → 2D
        reconstructed = self.decoder(latent)  # expand: 2D → 4D
        return reconstructed, latent   # return both (we want the latent space)

# ── Instantiate and train ──────────────────────────────────────
model = Autoencoder()
criterion = nn.MSELoss()              # MSE loss: how different is reconstruction from original?
optimizer = optim.Adam(model.parameters(), lr=0.001)
# Adam: adaptive learning rate optimiser — standard choice for neural networks
# model.parameters(): gives Adam all the weights it needs to update
# lr=0.001: learning rate (how big each weight update step is)

loss_history = []                      # track loss each epoch for plotting

for epoch in range(200):               # train for 200 full passes through the data
    total_loss = 0
    for batch_x, batch_y in loader:   # loop through mini-batches
        optimizer.zero_grad()         # reset gradients to zero (PyTorch accumulates them)
        reconstructed, _ = model(batch_x)  # forward pass: get reconstruction
        loss = criterion(reconstructed, batch_y)  # compare reconstruction to original
        loss.backward()              # backpropagation: compute gradients
        optimizer.step()             # update all weights one step downhill
        total_loss += loss.item()    # .item(): extract Python number from tensor
    loss_history.append(total_loss / len(loader))
    # average loss per epoch = total_loss / number of batches

print(f"Final reconstruction loss: {loss_history[-1]:.4f}")

# ── Extract latent space and visualise ────────────────────────
model.eval()                          # eval mode: disables dropout/batch norm effects
with torch.no_grad():                 # no_grad: don't compute gradients (saves memory)
    _, latent_vecs = model(X_tensor)  # run all data through, get latent representations
    latent_np = latent_vecs.numpy()  # convert PyTorch tensor → numpy array for plotting

fig, axes = plt.subplots(1, 2, figsize=(13, 5))

# Loss curve
axes[0].plot(loss_history, color='#7c3aed', linewidth=2)
axes[0].set_title('Autoencoder training loss')
axes[0].set_xlabel('Epoch')
axes[0].set_ylabel('Reconstruction Loss')
axes[0].grid(True, alpha=0.3)

# Latent space: compare to PCA
for i, sp_name in enumerate(species):

Part 7 · The Agentic AI Connection
Why this matters for AI Agents

Unsupervised learning is the invisible backbone of every AI agent you'll build

Every time you use an LLM in an agent, embeddings are involved. Embeddings are trained with contrastive learning — an unsupervised technique that pulls semantically similar text together in vector space and pushes dissimilar text apart. Same core idea as clustering. The 1536-number vector that represents a sentence in your RAG system came from an autoencoder-style neural network trained without any labels.

RAG (Retrieval Augmented Generation) — the technique used in practically every production AI agent — does semantic search over a vector database. Semantic search IS unsupervised learning: you're finding nearest neighbours in an embedding space. K-Means and hierarchical clustering are used under the hood to index and partition those vector databases for fast retrieval.

PCA and autoencoders are used to compress context. When an agent's context window starts filling up, compression models reduce conversation history to its essential meaning — the same bottleneck principle as PCA, implemented at scale. And Apriori-style pattern mining? It's the foundation of recommendation systems and tool-selection patterns in multi-step agent pipelines.
