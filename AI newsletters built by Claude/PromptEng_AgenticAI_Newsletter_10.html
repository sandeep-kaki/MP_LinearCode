Block 08
Prompt Engineering
Part 1 · Anatomy of a Good Prompt
The foundation

Anatomy of a good prompt — four parts, six rules

LLMs are sensitive to phrasing in ways that feel almost irrational until you understand why. The model has no persistent understanding of you, your context, or your intent. Every call is a blank slate. A well-structured prompt is what gives the model everything it needs to do what you actually want — nothing more, nothing less.

Every high-quality prompt has four parts. They don't all need to be present, but when a prompt fails, it's almost always because one of them is missing.

The four components of a production-grade prompt
SYSTEM PROMPT
Set the model's role, persona, rules, and constraints. Who is the model? What is it allowed to do? What must it never do? What format should it always use?
e.g. "You are a senior Python engineer. Always return code in a code block. Never suggest workarounds — only production-grade solutions. If you don't know, say so."
USER MESSAGE
The actual question or task. Be specific, explicit, and unambiguous. Vague requests produce vague responses — this is not a model limitation, it's an input problem.
e.g. "Write a function that takes a list of integers and returns them sorted in descending order, with duplicates removed."
CONTEXT / EXAMPLES
Background the model doesn't have, or demonstrations of what you want. Documents, data, few-shot examples, conversation history, or domain-specific facts not in the model's training data.
e.g. "Here is our internal API schema: [schema]. Here are two examples of correct responses: [example 1] [example 2]."
OUTPUT FORMAT
Exactly how the response should be structured. JSON schema, markdown headers, bullet points, a specific table format, a single word. If you need the output machine-readable, this is non-negotiable.
e.g. "Respond ONLY with a JSON object: {\"sentiment\": \"positive|negative|neutral\", \"confidence\": 0.0–1.0, \"reason\": \"one sentence\"}"

The six golden rules

RuleWhy it matters
Be specific and explicit"Summarise this" → 50% chance of the right length, tone, and depth. "Summarise in 3 bullet points, each under 20 words, for a non-technical executive audience" → consistent output every time.
Always specify output formatWithout a format, the model picks one. That choice may be inconsistent across calls and incompatible with your parsing code. JSON, markdown, table, single word — always say which.
Give context the model doesn't haveYour company's products, your user's history, today's date, your codebase conventions. The model can only work with what's in the context window. If it's not in the prompt, it doesn't exist.
Iterate — first prompt is rarely the bestPrompting is engineering. Version your prompts. Test them on 20–30 representative inputs. Measure which version fails. Fix the failures. The best prompts are built through iteration, not inspiration.
Tell it what NOT to doNegative constraints are often more reliable than positive ones. "Don't suggest workarounds" is clearer than "only suggest production solutions." Both together is best.
Test with edge casesThe happy path always works. Test: empty inputs, contradictory instructions, off-topic questions, adversarial inputs. Your prompt is only as good as its worst-case performance.
Part 2 · Basic Techniques
Core techniques

Six basic techniques — the daily vocabulary

Basic · 01
Zero-Shot
Just ask — no examples. "Translate to French: The weather is nice." The model draws on pre-training knowledge directly. Works for common, well-defined tasks the model has seen thousands of examples of during training.
When to use: Standard tasks — translation, summarisation, classification of common categories. When speed matters more than style consistency.
Basic · 02
Few-Shot
Give 2–5 input/output demonstrations before your actual query. The model recognises the pattern and applies it. More examples = more consistent output, but more tokens consumed.
When to use: When output format or style matters. Novel classification categories. Unusual writing styles. Any task where zero-shot produces inconsistent formatting.
Basic · 03
Role Prompting
"You are an expert [domain] with [years] of experience in [speciality]." Sets the model's perspective, vocabulary level, and depth of detail. Shifts tone from casual to professional or vice versa.
When to use: Technical depth ("expert Python engineer"), tone calibration ("patient teacher for beginners"), professional voice ("senior legal analyst").
Basic · 04
Instruction Prompting
Explicit, numbered step-by-step instructions. "Step 1: Extract the dates. Step 2: For each date, identify the event. Step 3: Return as a JSON array." Remove ambiguity by decomposing the task yourself.
When to use: Complex tasks the model keeps misunderstanding. Multi-step workflows where each step must happen in order.
Basic · 05
Negative Prompting
Explicitly tell the model what NOT to do or produce. "Do not include markdown formatting." "Never make up citations." "Do not ask clarifying questions — make your best attempt." Negative constraints are often more reliable than positive alternatives.
When to use: Removing unwanted default behaviours. Enforcing safety constraints. Preventing the model from adding unsolicited advice.
Basic · 06
Output Format Control
Specify exactly how the response must be structured: JSON schema, markdown headers, a table, a numbered list, a single word, or a specific XML structure. Critical for any production system that parses model output programmatically.
When to use: Any API integration. Structured data extraction. Anything downstream code will parse. Always.
Part 3 · Reasoning Techniques
Advanced techniques

Reasoning techniques — make the model think harder

The basic techniques tell the model what to produce. Reasoning techniques change how the model processes the problem before answering. The core insight: LLMs are better at problems when they reason through intermediate steps than when they jump straight to an answer. This isn't anthropomorphic — it's mechanistic. More tokens in the reasoning chain = more "computation" before the final answer token = higher accuracy on complex tasks.

"Think step by step" is perhaps the highest return-on-investment prompt modification that exists. Three words. Measurable accuracy improvement on every complex task.

Chain-of-Thought (CoT) Ask the model to reason step by step before giving a final answer. Technique: either include step-by-step reasoning examples in your few-shot demos, or explicitly say "Think through this step by step, then give your final answer." Why it works: the intermediate reasoning tokens create a "scratchpad" — the model can attend back to its own reasoning when generating the final answer, rather than needing to compute everything in one implicit forward pass. Google Brain's 2022 paper showed CoT dramatically improves accuracy on math, logic, and commonsense reasoning tasks.
Zero-Shot CoT The simplest form of CoT: append "Let's think step by step." to any prompt. No examples needed. This single phrase activates the model's reasoning mode and produces intermediate steps before the final answer. Works surprisingly well. Also effective: "Work through this carefully." / "Think about this systematically." / "What are all the considerations here?" Any phrase that signals "don't jump to the answer" triggers similar behaviour.
Self-Consistency Generate the same question multiple times with non-zero temperature (getting different reasoning chains each time), then take a majority vote over the final answers. For a math problem, generate 10 reasoning chains — if 7 arrive at "42", answer is "42" even if some intermediate steps differ. More reliable than single-shot for tasks with a correct answer. Costs more (10× the tokens and API calls). Best for high-stakes decisions where accuracy matters more than cost.
Prompt Chaining Break a complex task into multiple smaller prompts where the output of each becomes the input of the next. "Step 1 prompt: extract all entities → Step 2 prompt: for each entity, classify its type → Step 3 prompt: given the entity list, answer the question." Each step is simpler and more reliable than doing everything in one prompt. Also allows validation between steps — if step 1 output is wrong, you catch it before wasting tokens on steps 2 and 3. This is the foundation of agent pipeline design.
ReAct Prompting Reason + Act. A prompting pattern where the model alternates between Thought (reasoning about the current state), Action (calling a tool or taking a step), and Observation (seeing the result of the action). "Thought: I need to find the population of Chennai. Action: search[Chennai population 2024]. Observation: Chennai population is 7.1 million. Thought: Now I can answer." This is the foundational prompting pattern for AI agents — the Thought-Action-Observation loop is what enables a model to use tools iteratively to solve problems. Covered in depth in Block 09.
Tree of Thought (ToT) An extension of CoT: instead of one linear reasoning chain, explore multiple reasoning branches simultaneously and evaluate which branches look most promising before continuing. Think of it as beam search applied to reasoning rather than token generation. Prompt the model to generate several possible next reasoning steps, evaluate each, continue the most promising ones, backtrack from dead ends. More complex to implement (requires multiple LLM calls per reasoning step) but significantly outperforms CoT on tasks that require systematic exploration — puzzles, game-playing, multi-step planning. Intro concept — the principle matters more than the implementation at this stage.
The key insight — why reasoning techniques work

An LLM is a token predictor. Every token it generates is conditioned on all previous tokens. When you ask "What is 17 × 23?" the model must predict the answer in one shot from "?" → "391". When you ask "What is 17 × 23? Let's think step by step." the model generates "17 × 23 = 17 × 20 + 17 × 3 = 340 + 51 = 391." The intermediate tokens (340, 51) exist in the context and the model attends to them when generating "391". The correct answer is now the most likely next token because the reasoning was correct, not because the model memorised 17×23. This is why CoT helps on novel problems the model hasn't seen during training — it creates a computation path where memorisation is not required.

Part 4 · Pitfalls and Fixes
What goes wrong

Six common pitfalls — and exactly how to fix them

Every prompt engineer spends time debugging prompts that almost work. The same six failures appear over and over. Recognise the pattern and you can fix any prompt failure in minutes.

Problem What happens Fix
Vague instructions Model guesses at intent, produces output that's right shape but wrong content. "Summarise this" → you get a paragraph when you wanted bullet points. Be explicit with every dimension: length ("3 bullet points"), audience ("for a 10-year-old"), tone ("formal"), depth ("key numbers only, no narrative"). Leave nothing open to interpretation.
No output format Output varies between calls — sometimes markdown, sometimes prose, sometimes lists. Downstream parsing breaks on the inconsistent structure. Always specify format. For machine-readable: provide a JSON schema and say "respond ONLY with valid JSON, no other text." For human-readable: "use ## headers for each section, bullet points for lists."
Hallucinations Model confidently states false information — wrong dates, made-up citations, incorrect facts — because generating a plausible-sounding answer is easier than admitting uncertainty. Explicitly instruct: "If you are not certain, say 'I don't know' or 'I'm not sure.' Do not guess." For factual tasks, use RAG to ground the model in retrieved documents and instruct it to only answer from those documents.
Inconsistent results Same prompt, different outputs across calls in ways that matter for your application — different sentiment labels, different classifications, different styles. Add few-shot examples to anchor format and style. Lower temperature (0.1–0.3) for deterministic tasks. Add explicit definitions: "Positive = score 4-5/5, Neutral = score 3/5, Negative = score 1-2/5."
Ignores instructions Critical instructions buried in the middle of a long prompt are missed. The "lost-in-the-middle" problem: LLMs attend more strongly to content at the beginning and end of context. Put critical constraints at the start AND end of the system prompt. Use formatting to make them salient: "IMPORTANT: Never mention competitor products." Test specifically for instruction-following by crafting inputs designed to trigger the violation.
Too long prompt Unnecessary context dilutes attention on the parts that matter. More tokens = more cost = more latency. Model may lose track of specific constraints in a wall of text. Every sentence in a prompt should earn its place. Remove preamble, pleasantries, redundant explanations. Typical production system prompts: 200–600 words. If longer, modularise with clear XML tags: <rules>, <examples>, <format>.
The meta-fix: iterate with metrics

The fastest path to a reliable prompt is to build a test set of 20–50 representative inputs with expected outputs, then measure how often your current prompt gets the right answer. Every iteration should improve the score on the test set. Without measurement, prompt engineering is guesswork. With measurement, it's engineering. LangSmith and Braintrust are built specifically for this — they run your prompt against test cases automatically and show you which inputs fail.

If you're new: start with Anthropic Console or Google AI Studio (both free, no setup). Learn your way around system prompts, temperature, and few-shot examples. Block 09
Agentic AI — LLMs That Act in the World
Part 6 · What Makes an Agent
The definition

What makes an agent — five components

An LLM that answers questions is a tool. An AI agent is something different: it has a goal, it can take actions to achieve that goal, it observes the results of those actions, and it decides what to do next — repeatedly, until the goal is achieved. No human needs to guide each step.

The difference is not about the model — it's about the system around it. A GPT-4 answering questions is not an agent. The same GPT-4, given tools, a planning loop, and memory of what it's already tried, is an agent.

An agent is not a smarter chatbot. It's an LLM that has been given the ability to act — and a loop that keeps it acting until the job is done.

Component 1 · The Brain
LLM — Reasoning and Decision-Making
The LLM is the agent's brain. It reads the current state (task, memory, tool results), decides what to do next, and generates either a final answer or a tool call. Everything else is scaffolding — the LLM is the intelligence. This is why bigger, more capable models make better agents — they plan better, recover from errors more gracefully, and follow complex instructions more reliably.
Component 2 · The Hands
Tools — What the Agent Can Actually Do
Tools extend the LLM beyond text generation. Without tools, an agent can only think — it can't search, compute, or interact with external systems. With tools, it can check today's stock price, run Python code, query a database, send an email, or call any API. The set of tools is the set of actions available to the agent. Designing the right tools is one of the most important agent architecture decisions.
Component 3 · The Memory
Memory — What the Agent Knows and Has Done
Memory lets an agent accumulate context across steps (short-term) and across sessions (long-term). Without memory, each step is independent — the agent forgets what it tried, what failed, and what the user told it. With memory, it can learn from previous actions in the current task and from past interactions with the same user. Four types: short-term (in-context), long-term (vector DB), episodic (past conversations), procedural (successful strategies).
Component 4 · The Strategy
Planning — Breaking Goals into Steps
Given a complex goal ("research competitors and write a report"), planning is how the agent decomposes it into an ordered sequence of achievable sub-tasks: 1) identify competitors, 2) search for each competitor, 3) extract key info, 4) compare, 5) write report. Better planning = fewer dead ends, less wasted computation, more reliable task completion. This is where prompt engineering and agent design overlap — the planning quality depends heavily on how the task is framed to the LLM.
Component 5 · The Feedback Loop
Observation Loop — Act → See Result → Decide Next Step
The observation loop is what makes an LLM an agent instead of a pipeline. After every action, the agent observes the result and updates its understanding of the current state. Tool returned an error? Try a different approach. Search returned irrelevant results? Refine the query. Task completed successfully? Move to the next sub-task. This loop runs until the goal is achieved or a maximum step count is reached. The loop is typically: Thought → Action → Observation → Thought → Action → Observation → … → Final Answer.
Part 7 · Agent Patterns
How agents are structured

Five agent patterns — the architectures you'll actually build

ReAct — Reason + Act The most widely used agent pattern. The agent alternates between Thought (reasoning about what to do), Action (calling a tool with parameters), and Observation (seeing the tool's result). This loop repeats until the agent determines the task is complete and outputs a Final Answer. Simple to implement, interpretable (you can read the reasoning chain), and works reliably with capable LLMs. The foundational pattern — learn this before any other. Start here.

Plan-and-Execute The agent first generates a complete plan (ordered list of steps to achieve the goal), then executes each step one by one. Unlike ReAct which plans one step at a time, Plan-and-Execute commits to a full plan upfront. Advantage: better for long, structured tasks where the sequence of steps is mostly predictable and you want the plan to be reviewable before execution. Disadvantage: the plan can't adapt if early steps return unexpected results — less flexible than ReAct. Common hybrid: Plan-and-Execute with a ReAct loop inside each step execution.
Tool Use / Function Calling The LLM is given a list of available functions with their signatures and descriptions. When it needs to use one, it outputs a structured JSON object specifying the function name and arguments — not a text command, an actual machine-parseable call. The framework executes the function, injects the result back into the context, and the LLM continues. This is how modern agents are actually implemented: OpenAI's function calling API, Anthropic's tool use API, and LangChain's tool-calling abstractions all use this pattern. The LLM becomes a decision-maker about which function to call and with what parameters.
Multi-Agent A system where multiple specialised LLM agents collaborate on a task, each with different tools, knowledge, or responsibilities. An orchestrator agent receives the high-level goal, decomposes it, and delegates sub-tasks to specialised sub-agents (a researcher agent, a coder agent, a writer agent). Sub-agents complete their tasks and report back. More powerful than a single agent for complex, long-running tasks — but also more complex to build and debug. Covered in detail in the next section.
Human-in-the-Loop (HITL) A checkpoint system where the agent pauses and requests human approval or correction before taking certain actions. Can be configured at any granularity: "approve before any irreversible action" (deleting files, sending emails, making purchases), "approve before executing a plan" (show the plan, human edits or confirms), "approve after N steps" (let agent run but review every 5 steps). Essential for production agents where mistakes are costly or irreversible. Not a limitation — it's a design choice that makes agents trustworthy enough to deploy.
Start with ReAct — here's why

ReAct is the right starting pattern because it's interpretable: you can read the Thought-Action-Observation chain and understand exactly why the agent made each decision. When it fails, you see where. When Plan-and-Execute fails, you often only see the final wrong output. ReAct also adapts naturally to unexpected tool results — if the first search returns nothing, the next Thought can reformulate the query. Build your first five agents with ReAct before exploring more complex patterns.

Part 8 · Tool Use
Extending the LLM

Tool use — the six types of actions agents take

An LLM without tools is an oracle — you can ask it questions but it can't do anything. An LLM with tools is an agent — it can search the web, run code, query databases, call APIs, and manage files. The tools you give an agent define what problems it can solve. Here are the six tool categories every agent builder needs to know.

Tool TypeWhat it doesReal examplesWhen to include
Web Search Query the live internet and return current results. Solves the knowledge cutoff problem — agent can access information published after training. Tavily (built for LLM agents), SerpAPI (Google results), Bing Search API, DuckDuckGo API Any agent that needs current information: prices, news, recent events, live data.
Code Execution Run actual Python (or other language) code in a sandbox and return the output. Enables reliable computation, data analysis, and programmatic actions the LLM can't do reliably in its head. Python REPL (in-process), E2B sandbox (isolated cloud), Jupyter kernel, Docker container Any agent that does math, data analysis, file transformation, or needs to verify its own calculations.
Database Query Connect to a SQL or NoSQL database and retrieve structured data. The agent generates the query based on the user's natural language question (Text-to-SQL). SQL agent (LangChain), custom db tool, Vanna.ai, SQLite, PostgreSQL, BigQuery Any agent that needs to answer questions about structured internal data: sales, inventory, user records.
API Calls Call any external REST API and return the response. Connects the agent to virtually any external service. Weather API, stock price API, CRM (Salesforce), email (Gmail), calendar (Google Calendar), payment (Stripe), custom internal APIs Any agent that takes real actions in external systems — booking, sending, updating, fetching live data.
File Operations Read from and write to files — documents, PDFs, spreadsheets, CSVs, images. Enables document processing agents and file management workflows. PyPDF2 for PDFs, openpyxl for Excel, pandas for CSV, pathlib for filesystem, Google Drive API Any agent that processes uploaded documents, generates output files, or manages a file-based workflow.
Calculator Reliable arithmetic and mathematical computation. LLMs are notoriously unreliable at multi-digit arithmetic — a dedicated calculator tool ensures correct results. Python eval() (sandboxed), sympy, wolfram alpha API, custom math functions Any agent that does financial calculations, unit conversions, statistical computations, or any math where errors would be costly.
How function calling actually works under the hood

The agent framework serialises each tool as a JSON schema: name, description, parameters with types and descriptions. These schemas are injected into the LLM's system prompt or as a special API field. The LLM is instructed: "If you need to use a tool, respond with a JSON object in this format: {\"tool\": \"tool_name\", \"parameters\": {...}}." The framework parses this, calls the actual function, captures the return value, and injects it back as an Observation. The LLM never actually executes code itself — it only decides which function to call and with what parameters. The framework handles execution and safety.

Part 9 · Multi-Agent Systems
Scaling beyond one agent

Multi-agent systems — specialisation through collaboration

A single agent can handle tasks up to a certain complexity. Beyond that — very long tasks, tasks requiring deep expertise in multiple domains, tasks that can be parallelised — multi-agent systems outperform single agents. The pattern mirrors how teams of humans work: a manager delegates to specialists, specialists do their work and report back, the manager synthesises and delivers.

ConceptWhat it isWhy it matters
Orchestrator Agent The coordinator. Receives the high-level goal, decomposes it into sub-tasks, assigns each sub-task to the right specialised sub-agent, monitors progress, and synthesises the final result from all sub-agent outputs. Without an orchestrator, multi-agent systems have no coordination. The orchestrator is the only agent that sees the full goal — sub-agents only see their assigned slice. Requires a capable LLM (typically your best model) since it makes all strategic decisions.
Sub-Agents Specialised agents with a specific role, specific tools, and often a specific system prompt tuned for their domain. Common patterns: Researcher (web search + summarisation), Coder (code execution + code generation), Writer (document generation), Reviewer (fact-checking + critique), Data Analyst (database + charts). Specialisation improves quality. A coder agent can have a system prompt full of coding conventions and a Python execution environment. A researcher agent can have web search and document reading tools. Mixing all capabilities in one agent dilutes the system prompt and increases cognitive load on the LLM.
Agent Communication The mechanism by which agents share results, context, and partial outputs with each other. Typically implemented as messages in a shared conversation thread, or structured JSON objects passed between agent invocations. Communication determines whether the orchestrator can effectively synthesise sub-agent outputs. Structure matters: unstructured "I searched and found some things" is harder to use than {"sources": [...], "key_facts": [...], "confidence": 0.87}.
Human-in-the-Loop Checkpoints where a human reviews, corrects, or approves agent actions before they're executed or before the workflow continues. Can be event-driven ("ask before any file deletion") or time-based ("review after every 10 steps"). Critical for production multi-agent systems. Errors propagate — a wrong output from the researcher agent affects everything downstream. HITL at key checkpoints catches errors before they compound. Also essential for compliance and audit requirements in regulated industries.
The consulting firm analogy

A consulting firm receives a brief: "Analyse our market position and write a strategy report." The engagement manager (orchestrator) decomposes this into: market research (researcher agent), competitor analysis (researcher + data analyst agent), financial modelling (coder + data analyst agent), and report writing (writer agent). Each team works their slice, reports back to the engagement manager, who synthesises and delivers the final report. The client (human-in-the-loop) reviews the draft before it's finalised. This is a multi-agent system. Your code just runs faster and costs less than an actual consulting firm.

Part 10 · Agent Memory Types
What agents remember

Four memory types — how agents accumulate knowledge

An LLM has no state between API calls. Every new call starts fresh. Memory systems are what give agents continuity — the ability to remember what they've done, what the user told them, and what strategies have worked before. Four distinct memory types serve different purposes.

Memory Type 1
Short-Term (In-Context)
Everything in the current context window — the task description, all previous steps, all tool outputs, all observations so far in this session. Automatically available to the LLM since it's in the prompt. Limited by context window size. When the window fills, oldest content is dropped.
Example: In a 10-step ReAct loop, the LLM sees all 9 previous Thought-Action-Observation triplets when generating step 10. No special implementation needed — it's just in the prompt.
Memory Type 2
Long-Term (External)
Facts, preferences, and past session summaries stored in an external database (typically a vector database) and retrieved at the start of each new session based on relevance to the current task. Persists across conversations. Enables the agent to remember you across sessions.
Example: User says "I prefer Python over JavaScript." Agent writes this to vector DB. Three sessions later, when discussing a new task, the agent queries memory and retrieves the language preference before suggesting code examples.
Memory Type 3
Episodic
Stored records of specific past interactions — whole conversations or key moments — recalled by similarity to the current situation. "I've handled a request like this before." The agent retrieves the past episode as context for handling the current one.
Example: A support agent remembers how it resolved a specific billing issue two months ago and retrieves that episode when a similar billing complaint arrives. The past resolution becomes context for the current case.
Memory Type 4
Procedural
Stored strategies, plans, and workflows that worked in the past. "Here's how to handle this type of task." Retrieved when a similar task is encountered and used to initialise the agent's planning rather than planning from scratch every time.
Example: An agent learns that "research + outline + draft + review" is a reliable workflow for writing technical articles. It stores this sequence as a procedural memory and retrieves it whenever a writing task begins, skipping the planning step.
Libraries for agent memory

LangChain Memory: ConversationBufferMemory (store full history), ConversationSummaryMemory (summarise old turns), ConversationVectorStoreMemory (store and retrieve by similarity). LangGraph state: Persistent state graph that carries agent context across nodes — the cleanest solution for complex multi-step agents. LlamaIndex: Document and conversation indexing for long-term retrieval-based memory. mem0: Dedicated memory layer for AI agents — automatic extraction and retrieval of user preferences and past interactions.

# ── CoT vs Direct: same question, different prompting ────────
print("=== Chain-of-Thought vs Direct Answer ===\n")

problem = """A store sells notebooks for ₹45 each and pens for ₹12 each. 
Rohan buys 7 notebooks and 14 pens. He pays with a ₹500 note. 
How much change does he receive?"""

# ── Approach 1: Direct (no CoT) ───────────────────────────────
direct_response = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=50,
    messages=[{"role": "user",
               "content": problem + "\n\nAnswer with only the number of rupees change."}]
)
print("DIRECT (no CoT):")
print(f"Answer: {direct_response.content[0].text.strip()}")

# ── Approach 2: Zero-shot CoT ─────────────────────────────────
cot_response = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=300,
    messages=[{"role": "user",
               "content": problem + "\n\nThink step by step, then give the final answer."}]
    # "Think step by step" is the zero-shot CoT trigger
)
print("\nCHAIN-OF-THOUGHT:")
print(cot_response.content[0].text.strip())

print("\n--- Correct answer: ₹131 ---")
print("(7×45 = 315, 14×12 = 168, 315+168 = 483, 500-483 = 17 ... wait: ₹17)")
print("Actually: 7×45=315, 14×12=168, total=483, change=500-483=₹17")
Expected Output
DIRECT (no CoT):
Answer: ₹17

CHAIN-OF-THOUGHT:
Let me work through this step by step.

Step 1: Cost of notebooks
7 notebooks × ₹45 = ₹315

Step 2: Cost of pens
14 pens × ₹12 = ₹168

Step 3: Total cost
₹315 + ₹168 = ₹483

Step 4: Change received
₹500 - ₹483 = ₹17

Rohan receives ₹17 in change.

--- Correct answer: ₹17 ---
→ Both got it right here — simple enough problem for direct answer
→ CoT advantage shows up on multi-step reasoning problems with 5+ steps
→ The intermediate tokens (315, 168, 483) give the model a "scratchpad"
→ CoT uses 6× more tokens — worth it for complex problems, overkill for simple ones
→ For complex logic/math: always use CoT. For classification/extraction: direct is fine.

Cell 4 — ReAct agent loop from scratch (no framework)
import json, re
from datetime import datetime

# ── Define tools (simulated — no real API keys needed) ─────────
def web_search(query: str) -> str:
    """Simulate web search — returns mock results."""
    mock_results = {
        "Chennai population":       "Chennai population (2024): approximately 7.1 million in city, 11.2 million metro area.",
        "Python release date":       "Python 3.12 released October 2023. Python 3.13 released October 2024.",
        "India GDP 2024":            "India GDP 2024: approximately $3.7 trillion USD, 7th largest economy globally.",
    }
    for key, result in mock_results.items():
        if key.lower() in query.lower():
            return result
    return f"No specific results found for '{query}'. Try a different query."

def calculator(expression: str) -> str:
    """Safe calculator — evaluates arithmetic expressions."""
    try:
        result = eval(expression, {"__builtins__": {}}, {})
        # eval with empty builtins — safe for arithmetic only
        return f"Result: {result}"
    except Exception as e:
        return f"Error: {e}"

def get_date() -> str:
    return f"Current date: {datetime.now().strftime('%B %d, %Y')}"

TOOLS = {
    "web_search": (web_search, "Search the web for current information. Input: search query string."),
    "calculator": (calculator, "Calculate arithmetic. Input: Python arithmetic expression (e.g. '7 * 45 + 14 * 12')."),
    "get_date":   (get_date,   "Get the current date. No input needed."),
}

# ── Build the ReAct system prompt ─────────────────────────────
tools_desc = "\n".join([f"- {name}: {desc}" for name, (fn, desc) in TOOLS.items()])
SYSTEM = f"""You are a helpful AI agent. You solve problems step by step using tools.

Available tools:
{tools_desc}

Respond using EXACTLY this format on each turn:
Thought: [your reasoning about what to do next]
Action: [tool_name]
Action Input: [input to the tool]

OR, when you have the final answer:
Thought: [your final reasoning]
Final Answer: [your complete answer to the user]

Never skip the Thought. Never output anything else."""

# ── Run the ReAct loop ─────────────────────────────────────────
def run_react_agent(task: str, max_steps: int = 6) -> str:
    messages = [{"role": "user", "content": task}]
    print(f"Task: {task}\n" + "-" * 55)

    for step in range(max_steps):
        response = client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=400,
            system=SYSTEM,
            messages=messages
        )
        agent_output = response.content[0].text.strip()
        print(f"\nStep {step+1}:\n{agent_output}")

        # check if agent reached final answer
        if "Final Answer:" in agent_output:
            return agent_output.split("Final Answer:")[1].strip()

        # parse Action and Action Input
        action_match = re.search(r'Action:\s*(\w+)', agent_output)
        input_match  = re.search(r'Action Input:\s*(.+?)(?=\n|$)', agent_output, re.DOTALL)

        if action_match:
            action_name = action_match.group(1).strip()
            action_input = input_match.group(1).strip() if input_match else ""

            if action_name in TOOLS:
                tool_fn, _ = TOOLS[action_name]
                observation = tool_fn(action_input) if action_input else tool_fn()
                print(f"Observation: {observation}")
                # inject tool result back as an assistant-turn + observation
                messages.append({"role": "assistant", "content": agent_output})
                messages.append({"role": "user", "content": f"Observation: {observation}"})
            else:
                messages.append({"role": "assistant", "content": agent_output})
                messages.append({"role": "user", "content": f"Observation: Tool '{action_name}' not found."})
    return "Max steps reached without final answer."

# ── Run it ─────────────────────────────────────────────────────
answer = run_react_agent("What is Chennai's population, and what percentage of India's total population (1.44 billion) does that represent? Round to 2 decimal places.")
print(f"\n{'='*55}\nFINAL ANSWER: {answer}")

# ── Anthropic's native tool use API ───────────────────────────
print("=== Anthropic Native Tool Use (Function Calling) ===\n")

# Define tools as JSON schemas — the model reads these to know what's available
tools = [
    {
        "name": "get_weather",
        "description": "Get the current weather for a city. Returns temperature and conditions.",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "City name"},
                "unit": {"type": "string", "enum": ["celsius", "fahrenheit"], "description": "Temperature unit"}
            },
            "required": ["city"]
        }
    },
    {
        "name": "calculate",
        "description": "Perform arithmetic calculations reliably.",
        "input_schema": {
            "type": "object",
            "properties": {
                "expression": {"type": "string", "description": "Python arithmetic expression"}
            },
            "required": ["expression"]
        }
    }
]

# ── Simulated tool execution ──────────────────────────────────
def execute_tool(name: str, inputs: dict) -> str:
    if name == "get_weather":
        city = inputs.get("city", "Unknown")
        unit = inputs.get("unit", "celsius")
        mock = {"Chennai": (32, "sunny and humid"), "Mumbai": (29, "partly cloudy"), "Delhi": (25, "clear skies")}
        temp, cond = mock.get(city, (22, "unknown"))
        if unit == "fahrenheit": temp = temp * 9/5 + 32
        return f"{city}: {temp}°{'F' if unit=='fahrenheit' else 'C'}, {cond}"
    elif name == "calculate":
        try: return f"= {eval(inputs['expression'], {'__builtins__': {}}, {})}"
        except Exception as e: return f"Error: {e}"
    return "Tool not found"

# ── Run with Anthropic's tool use API ────────────────────────
messages = [{"role": "user", "content": "What's the weather in Chennai and Mumbai? Also, what's the average of their temperatures?"}]

while True:
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=500,
        tools=tools,
        messages=messages
    )

    if response.stop_reason == "end_turn":
        break   # model decided it's done

    if response.stop_reason == "tool_use":
        # model wants to call tools — find all tool_use blocks
        messages.append({"role": "assistant", "content": response.content})
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                print(f"Calling: {block.name}({block.input})")
                result = execute_tool(block.name, block.input)
                print(f"Result:  {result}\n")
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,   # must match the tool_use ID
                    "content": result
                })
        messages.append({"role": "user", "content": tool_results})

final_text = "".join(b.text for b in response.content if hasattr(b, 'text'))
print(f"Final: {final_text}")
