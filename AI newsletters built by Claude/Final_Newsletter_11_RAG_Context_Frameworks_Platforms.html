Block 10
RAG — Retrieval-Augmented Generation
Part 1 · Why RAG
The most important pattern in production AI

Why RAG — five problems, one architecture

An LLM is frozen in time. It knows everything that was in its training data — and nothing else. No events after its cutoff. No private company documents. No real-time prices or news. RAG (Retrieval-Augmented Generation) breaks this constraint by giving the model access to a dynamic, queryable knowledge base at inference time.

RAG doesn't change the model. It changes what the model can see. And the model can only reason about what it can see.

ProblemWhat happens without RAGWhat RAG does
Knowledge cutoff LLM doesn't know about events, products, or documents published after its training cutoff date. Retrieve live documents at query time — web pages, PDFs, news articles — and inject them as context. The model answers from retrieved facts, not stale training memory.
Private data LLM was never trained on your company's internal documentation, customer data, or proprietary knowledge base. Index your private documents in a vector database. Retrieve the relevant chunks when a query arrives. No model retraining needed — the knowledge is in the retrieval layer.
Hallucination LLM confidently states incorrect facts when it doesn't know the answer, rather than admitting uncertainty. Ground the model in retrieved documents. Instruct it to answer only from the provided context. Hallucination rate drops dramatically when the model has real facts to work with.
Long documents A 500-page manual doesn't fit in the context window. Even if it did, important details would be lost in the middle. Chunk the document, embed all chunks, store in vector DB. At query time, retrieve only the 3–5 most relevant chunks. The model reads 500 words instead of 500 pages.
Cost Fine-tuning a model on new knowledge requires GPU compute, data prep, and retraining — expensive and slow to update. Adding new knowledge = adding documents to the vector database. No retraining. No compute cost beyond the embedding pass. Update knowledge in minutes.

Part 3 · Key Concepts
What every RAG engineer needs to know

Key concepts — the details that determine quality

Chunking Strategies How you split documents determines retrieval quality. Fixed-size chunking: split every N tokens (e.g. 512), with overlap of ~50 tokens between chunks to preserve context at boundaries. Simple but loses semantic coherence. Sentence chunking: split on sentence boundaries — better semantic units but variable size. Semantic chunking: use an embedding model to detect where topic shifts occur and split there — best quality, highest cost. Recursive character splitting (LangChain default): hierarchically try to split on paragraph → sentence → word boundaries, falling back down the hierarchy. Rule of thumb: chunk size should match your query granularity — question about one fact → small chunks (256–512 tokens); question about a topic → larger chunks (1024–2048 tokens).
Embedding Models The model that converts text to vectors. The same model must be used for both documents (at indexing) and queries (at retrieval) — otherwise vectors live in different spaces and similarity is meaningless. Key options: text-embedding-3-small (OpenAI, 1536 dims, fast, cheap, excellent quality), text-embedding-3-large (3072 dims, highest quality), BGE-M3 (open-source, multilingual, strong performance), E5-large (open-source, strong on retrieval tasks), sentence-transformers (Python library, many models, local). Dimension count ≠ quality — BGE-M3 at 1024 dims often beats older models at 1536 dims.
Top-K Retrieval Return the K most similar chunks to the query. K=3 is a common default. K=5–10 for complex questions that may need multiple evidence pieces. Higher K = more context = better recall but more tokens and more noise in the context. The challenge: a high similarity score doesn't always mean relevant content — a chunk about "bank interest rates" may score high for "river bank" queries if the domain isn't controlled. This is why reranking exists.
Reranking A second-pass model that takes the top-K retrieved chunks and the original query, and re-scores them for true relevance — not just vector similarity. First retrieval (vector search) is fast but approximate. Reranking is slower but more precise. The standard pattern: retrieve top-20 with vector search (high recall), then rerank and keep top-3 (high precision). Popular rerankers: Cohere Rerank API, cross-encoder models (ms-marco-MiniLM-L-6-v2), BGE-Reranker. Reranking consistently improves RAG answer quality with modest additional cost.
Hybrid Search Combine dense vector search (semantic similarity) with sparse keyword search (BM25/TF-IDF) and merge the results. Vector search excels at semantic matching ("how do I fix a broken authentication flow?"). Keyword search excels at exact term matching (product codes, names, abbreviations, proper nouns). Hybrid search beats either alone on real-world queries which often contain both semantic intent and specific terms. RRF (Reciprocal Rank Fusion) is the standard algorithm for merging ranked lists from both approaches. Elasticsearch (8.x+), Pinecone, Weaviate, and Qdrant all support hybrid search natively.
Part 4 · RAG Types
From basic to production-grade

Four RAG types — an evolution path

Type 1 · Start Here
Naive RAG
The basic pipeline: ingest → chunk → embed → store → retrieve top-K → inject → generate. No reranking, no query transformation, no advanced chunking. Surprisingly effective for well-structured documents. The right starting point — understand its failure modes before reaching for advanced techniques.
Type 2 · Better Quality
Advanced RAG
Adds pre-retrieval improvements (query rewriting, query expansion, HyDE — hypothetical document embedding) and post-retrieval improvements (reranking, context compression, result fusion). Typically 20–40% better answer quality on real-world queries. Use when Naive RAG fails consistently on a specific failure mode.
Type 3 · Production Flexibility
Modular RAG
Decouple every pipeline component into swappable modules: any loader, any chunker, any embedder, any retriever, any reranker, any LLM, any output parser. Mix and match based on the task. LlamaIndex and LangChain both support this architecture. Easier to debug, A/B test, and upgrade individual components without rebuilding the whole pipeline.
Type 4 · Agent-Powered
Agentic RAG
An agent dynamically decides when to retrieve, what to retrieve, and whether to retrieve again based on the current answer quality. If the first retrieval doesn't produce a confident answer, the agent reformulates the query and tries again. Combines the ReAct pattern with RAG — each "Action" can be a retrieval call. Most powerful but most complex. Use when queries are complex, ambiguous, or multi-hop (require information from multiple documents).
RAG libraries quick reference

LangChain: End-to-end RAG with any vector store, any LLM, any loader. Best for general-purpose RAG as part of a larger agent. LlamaIndex: Document-focused RAG with sophisticated chunking, indexing strategies, and query engines. Best when your application is primarily about querying your own documents. ChromaDB: Lightweight, runs locally, zero infrastructure setup — perfect for development and small-scale production. FAISS: Facebook's ultra-fast vector search for large-scale local deployment — no network latency, runs in-process. Pinecone: Fully managed cloud vector database — scaling, persistence, backups handled for you. HuggingFace sentence-transformers: Open-source embedding models library — run any SBERT-compatible model locally at zero API cost.

Block 11
Context Engineering — What the Model Can See
Part 5 · The Context Window
The most valuable real estate in AI

The context window — your model's entire world

An LLM can only reason about what is currently in its context window. This is not a limitation of intelligence — it is a structural constraint. If the relevant document isn't in the context, the model doesn't know it exists. If the key instruction isn't in the prompt, the model doesn't know to follow it. Context engineering is the discipline of making sure the right information is always in the window, in the right place, in the right form.

ConceptWhat it means in practice
Context window The maximum number of tokens the model can process in one call — both input (prompt + documents + history) and output (the generated response) count against this limit.
Tokens How text is measured for billing and limits. Roughly 1 word ≈ 1.3 tokens in English. 1 token ≈ 4 characters. Non-English languages are less token-efficient (Tamil: ~3 chars/token). Code is typically more token-efficient than prose.
Context limit growth GPT-2 (2019): 1,024 tokens. GPT-3 (2020): 4,096. GPT-4 (2023): 8K–128K. Claude 3.5 (2024): 200,000. Gemini 1.5 Pro: 1,000,000. The growth has been exponential — but cost scales with context length, so longer isn't always better.
What fits inside System prompt + conversation history + retrieved documents + few-shot examples + the current user query + the model's response. All of these compete for the same token budget. Context engineering is resource allocation for tokens.

The lost-in-the-middle problem — a critical practical insight

Research (Liu et al., Stanford 2023 — "Lost in the Middle") showed that LLMs are significantly better at using information placed at the beginning or end of a long context than information buried in the middle. When the same relevant document was placed at position 1 (start), position 15 (middle), or position 30 (end) of a 30-document context, accuracy was highest at the extremes and lowest in the middle.

Practical implications of lost-in-the-middle

In RAG: put the most relevant retrieved chunk first, not buried behind less relevant ones. In system prompts: put the most critical constraints at the top AND repeat the most important rule at the bottom. In conversation history: recent messages matter more — compress or drop old turns rather than keeping them all equally. In document injection: use XML labels (<document id="1" relevance="high">) to help the model identify the most relevant content without relying on position alone.

Part 6 · Memory Systems + Context Strategies
How agents remember across calls

Memory systems — five types, five purposes

Every LLM call starts with a blank slate — no memory of previous calls. Memory systems are how you give applications continuity. Context engineering determines which memories get injected into each call, how much space they're allocated, and in what form.

Memory TypeHow it worksWhen to use
In-context memory All previous turns in the current conversation, already in the prompt. Automatic — no special implementation. Limited by context window: a 200K window fits ~150,000 words of history. Short conversations, single-session tasks. The default for all chatbots.
External memory (RAG) Stored in a vector database, retrieved on demand based on relevance to the current query. Unlimited scale — can store millions of documents. Only the relevant ones are retrieved per call. Long-term knowledge storage, private documents, any knowledge base larger than the context window.
Episodic memory Stored summaries or records of specific past conversations, recalled by similarity to the current session. "I've handled a request like this before — here's what I did." Support agents that learn from past cases. Personal assistants that remember previous interactions with a user.
Semantic memory Extracted facts, preferences, and knowledge stored as searchable embeddings. "User prefers Python. User works at a fintech startup. User is at an intermediate skill level." Personalised agents that adapt to individual users over time. Knowledge accumulation from conversations.
Working memory Temporary notes, intermediate results, and scratchpad reasoning written into the prompt context during the current task. "Note: the user mentioned a budget of ₹50,000." Complex multi-step tasks where intermediate conclusions need to be tracked within a single session.
When the window fills up

Context management strategies — five ways to manage the budget

StrategyHow it worksBest for
Sliding window Keep only the most recent N tokens of conversation history. When the window would overflow, drop the oldest messages. Simple, fast, no additional LLM calls needed. Chat applications where very old conversation turns are rarely relevant. Poor for tasks where early context is critical (e.g. "remember the goal I stated at the start").
Summarisation compression When old conversation turns would overflow the window, use the LLM to summarise them into a compact paragraph ("Earlier, the user explained X and you agreed to do Y"). Inject the summary instead of the full history. Long conversations where the overall narrative matters but exact wording doesn't. Adds latency and cost of a summarisation call but preserves semantic content.
Selective retrieval (RAG) Don't load all documents upfront. At each query, retrieve only the 3–5 chunks relevant to that specific query. The context window contains targeted relevant content rather than a dump of everything. Document Q&A, knowledge base agents, any application where the knowledge base is larger than a few thousand tokens.
Context prioritisation Assign importance scores to different context elements. In a crisis, keep system prompt + most recent messages + most relevant documents. Drop: old conversation, low-relevance retrieved chunks, verbose examples. Complex agents with many context sources competing for space. Requires explicit token budget management.
Document chunking Break large documents into smaller, semantically coherent pieces before storing or injecting. A 50-page PDF becomes 200 chunks of 512 tokens each. Insert only the relevant chunks, not the full document. Any application involving documents longer than 2,000 tokens. Essential for production RAG systems.
The architecture of a well-engineered context

Structured context design — how to organise what goes in

The newspaper layout analogy

A newspaper puts the most important headline above the fold — before you unfold the paper, you've already seen the key story. Every other item is ordered by importance, not randomly. Context engineering is the same: the model reads top-to-bottom, attention weights certain positions more heavily (start and end), and your most critical content must be placed deliberately — not wherever it happened to end up.

TechniquePurposeExample
XML / Markdown tagging Clearly label and delineate different sections so the model can identify what type of content each section contains and treat it appropriately. <documents>, <examples>, <question>, <conversation_history>
Grounding Explicitly instruct the model to only use the provided context for its answer, reducing reliance on training-time knowledge and hallucination. "Answer using ONLY the documents provided below. If the answer is not in the documents, say 'I don't have that information.'"
Persona + constraints first Front-load the system prompt with the most critical instructions. What the model reads first shapes how it processes everything that follows. Role definition → absolute rules → output format → then everything else
Document injection Insert retrieved documents with clear labels showing their source, relevance rank, and scope. Don't dump raw text — structure it so the model knows what it's reading. <doc id="1" source="HR_Policy.pdf" pages="12-13">...content...</doc>

The ideal context structure — five positions

Ideal context window order — top to bottom
1
System prompt + persona + rules Who the model is, what it must always do, what it must never do, absolute constraints, output format. These are read first and shape everything. Put critical rules here AND repeat the most important at position 5.
2
Retrieved documents / grounding facts The specific knowledge retrieved from your vector database for this query. Near the top = strong attention. Label with XML tags. Include source attribution.
3
Conversation history (compressed) Summary of earlier turns in this session. Summarised, not verbatim. Important events preserved; exact wording compressed. Gives the model awareness of what's been discussed.
4
Few-shot examples (near the query) Input/output examples that demonstrate the desired format. Placed just before the query because the model will directly pattern-match the examples to the upcoming task.
5
User query (last — model reads it fresh) The actual question or task — placed last so the model has all context when it processes it. Models attend more strongly to the end of context. Repeat any critical constraint here too: "Remember: always respond in JSON."

Cell 1 — Complete RAG pipeline from scratch: chunk → embed → store → retrieve → generate
import subprocess
subprocess.run(["pip","install","sentence-transformers","faiss-cpu","anthropic","-q"])
from sentence_transformers import SentenceTransformer
import faiss, numpy as np, anthropic, os

# ── PHASE 1: INDEXING ─────────────────────────────────────────
# In production: load PDFs/Word docs. Here: inline text documents.

documents = [
    {"id": 1, "text": "Python is a high-level, interpreted programming language created by Guido van Rossum in 1991. It emphasises code readability and supports multiple paradigms including procedural, object-oriented, and functional programming.", "source": "python_intro.txt"},
    {"id": 2, "text": "Machine learning is a subset of AI where algorithms learn patterns from data without explicit programming. Key algorithms include linear regression, decision trees, neural networks, and support vector machines.", "source": "ml_basics.txt"},
    {"id": 3, "text": "The Transformer architecture, introduced in 'Attention Is All You Need' (2017), uses self-attention mechanisms to process sequences in parallel. It is the foundation of models like BERT, GPT, and Claude.", "source": "transformer.txt"},
    {"id": 4, "text": "RAG stands for Retrieval-Augmented Generation. It combines a retrieval system with a language model to answer questions using external knowledge not in the model's training data.", "source": "rag_overview.txt"},
    {"id": 5, "text": "LangChain is a framework for building LLM applications. Its core components are LLMChain, PromptTemplate, OutputParser, Memory, Retriever, and Agent+Tools.", "source": "langchain.txt"},
    {"id": 6, "text": "Chennai, officially known as Madras until 1996, is the capital of Tamil Nadu, India. It is the fourth-largest city in India with a population of approximately 7.1 million.", "source": "chennai.txt"},
]

# ── Step 2: Chunking (documents are small, so no splitting needed here) ──
# In production: RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=50)
texts = [d["text"] for d in documents]
sources = [d["source"] for d in documents]

# ── Step 3: Embedding ─────────────────────────────────────────
embed_model = SentenceTransformer("all-MiniLM-L6-v2")
# same model must be used for documents AND queries
embeddings = embed_model.encode(texts, convert_to_numpy=True).astype(np.float32)
faiss.normalize_L2(embeddings)        # L2 norm → inner product = cosine similarity

# ── Step 4: Vector store ─────────────────────────────────────
index = faiss.IndexFlatIP(embeddings.shape[1])
index.add(embeddings)
print(f"✅ Indexed {index.ntotal} documents ({embeddings.shape[1]}-dim vectors)\n")

# ── PHASE 2: RETRIEVAL + GENERATION ──────────────────────────
client = anthropic.Anthropic(api_key=os.environ.get("ANTHROPIC_API_KEY"))

def rag_query(question: str, k: int = 3) -> str:
    # Step 5: Query embedding
    q_emb = embed_model.encode([question], convert_to_numpy=True).astype(np.float32)
    faiss.normalize_L2(q_emb)

    # Step 6: Similarity search — top-k
    scores, indices = index.search(q_emb, k)
    retrieved = [(texts[i], sources[i], scores[0][j]) for j, i in enumerate(indices[0])]

    # Step 7: Context injection — format retrieved chunks as labelled XML
    context = "\n".join([
        f'<document id="{j+1}" source="{src}" relevance="{score:.3f}">\n{text}\n</document>'
        for j, (text, src, score) in enumerate(retrieved)
    ])

    # Step 8: LLM generation grounded in retrieved context
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=300,
        system="""You are a helpful assistant. Answer questions using ONLY the provided documents.
If the answer is not in the documents, say: "I don't have that information in my knowledge base."
Always cite the source document in your answer.""",
        messages=[{"role": "user", "content": f"Documents:\n{context}\n\nQuestion: {question}"}]
    )
    return response.content[0].text, retrieved

print("=== RAG Query Results ===\n")
queries = [
    "What is the Transformer architecture and who created it?",
    "What is the population of Chennai?",
    "What is the GDP of India?",  # not in any document — tests grounding

# ── Strategy 1: Sliding window ─────────────────────────────────
TOKEN_BUDGET = 200   # example budget for conversation history
def sliding_window(messages, budget):
    """Keep most recent messages that fit within token budget."""
    result, tokens_used = [], 0
    for msg in reversed(messages):      # most recent first
        t = count_tokens(msg["content"])
        if tokens_used + t <= budget:
            result.insert(0, msg)
            tokens_used += t
        else:
            break
    return result, tokens_used

Cell 5 — Complete production RAG with chunking + reranking + structured output
import subprocess
subprocess.run(["pip","install","sentence-transformers","faiss-cpu","-q"])
from sentence_transformers import SentenceTransformer, CrossEncoder
import faiss, numpy as np, json

# ── Simulate a larger document corpus ─────────────────────────
long_document = """
Section 1 - Introduction to RAG:
RAG stands for Retrieval-Augmented Generation. It was introduced to solve LLM knowledge cutoff and hallucination problems.

Section 2 - Chunking Strategies:
Fixed-size chunking splits every N tokens. Semantic chunking splits on topic boundaries. Sentence chunking splits on punctuation.

Section 3 - Vector Databases:
FAISS is a local, fast vector database by Meta. Pinecone is a managed cloud vector database. ChromaDB is lightweight and perfect for development.

Section 4 - Embedding Models:
text-embedding-3-small from OpenAI produces 1536-dimensional vectors. BGE-M3 is an open-source multilingual embedding model. E5-large is strong for retrieval tasks.

Section 5 - Reranking:
Reranking uses a cross-encoder model to re-score retrieved chunks by relevance. Retrieve top-20 with vector search, rerank, keep top-3. Improves precision significantly.

Section 6 - Hybrid Search:
Hybrid search combines dense vector search with sparse BM25 keyword search. It uses Reciprocal Rank Fusion (RRF) to merge both ranked lists. Consistently beats either approach alone.

Section 7 - RAG Types:
Naive RAG: basic retrieve-inject-generate. Advanced RAG: query rewriting, reranking. Modular RAG: swappable components. Agentic RAG: agent decides when and what to retrieve.
"""

# ── Chunking: fixed size with overlap ─────────────────────────
def chunk_text(text: str, chunk_size: int = 120, overlap: int = 20) -> list:
    """Split text into overlapping word chunks."""
    words = text.split()
    chunks = []
    for i in range(0, len(words), chunk_size - overlap):
        chunk = " ".join(words[i:i + chunk_size])
        if chunk.strip():
            chunks.append(chunk)
    return chunks

chunks = chunk_text(long_document)
print(f"Document → {len(chunks)} chunks (chunk_size=120 words, overlap=20)\n")

# ── Embed all chunks ──────────────────────────────────────────
embed_model = SentenceTransformer("all-MiniLM-L6-v2")
chunk_embeddings = embed_model.encode(chunks, convert_to_numpy=True).astype(np.float32)
faiss.normalize_L2(chunk_embeddings)
index = faiss.IndexFlatIP(chunk_embeddings.shape[1])
index.add(chunk_embeddings)

# ── Reranking with CrossEncoder ───────────────────────────────
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")
# CrossEncoder: takes (query, passage) pairs and scores true relevance
# More accurate than cosine similarity but slower (can't pre-compute)

query = "What is the difference between vector search and hybrid search in RAG?"

# Step 1: Retrieve top-10 with vector search (high recall)
q_emb = embed_model.encode([query], convert_to_numpy=True).astype(np.float32)
faiss.normalize_L2(q_emb)
scores, indices = index.search(q_emb, k=10)
top10_chunks = [chunks[i] for i in indices[0]]

# Step 2: Rerank top-10 → keep top-3 (high precision)
rerank_scores = reranker.predict([(query, c) for c in top10_chunks])
# reranker.predict: feeds (query, chunk) pairs, outputs relevance scores
ranked = sorted(zip(rerank_scores, top10_chunks), reverse=True)[:3]

print(f"Query: '{query}'\n")
print("Vector search top-3 → Reranked top-3:")
for rank, (score, chunk) in enumerate(ranked, 1):
    preview = chunk[:80].replace('\n', ' ')
    print(f"  Rank {rank} (rerank score {score:.3f}): '{preview}...'")

print(f"\nPattern: retrieve top-20 → rerank → keep top-3 = higher precision at low cost")
