Block 06
Language Models — SLMs & LLMs
Part 1 · Pre-Training
Block 06 · Section 1
Pre-Training — Learning Everything by Predicting the Next Word
Train a Transformer decoder on trillions of tokens with one task: predict the next token. No labels needed. As a side effect, the model learns the entire structure of human knowledge.
Task: Next Token Prediction (CLM)
Data: Common Crawl · Books · Code · Wikipedia
Pre-Training

Next Token Prediction — the one task that teaches everything

The pre-training objective sounds almost trivially simple: given a sequence of tokens, predict the next one. "The cat sat on the ___" → predict "mat". That's it. There's no labelled data, no human annotation, no explicit teaching of grammar or facts or reasoning. Just: predict the next token, measure error, backpropagate, repeat.

But here's the thing — to reliably predict the next token across trillions of examples, a model must implicitly learn everything. It must learn grammar (wrong grammar → wrong predictions). It must learn facts (predicting "France" after "The capital of France is" requires knowing the fact). It must learn reasoning (predicting the conclusion of a logical argument requires following the logic). All of this emerges as a side effect of next-token prediction at scale.

Language modelling is not a task. It's a compression challenge — the only way to predict text reliably across all of human knowledge is to genuinely model the world that produced it.

BPE — Byte Pair Encoding The most widely used tokenisation algorithm for LLMs. Start with individual characters. Find the most frequent pair of adjacent characters or tokens and merge them into one new token. Repeat thousands of times. Result: a vocabulary of ~50K subword tokens. "unhappiness" becomes ["un", "happi", "ness"]. "GPT-4" becomes ["G", "PT", "-", "4"]. BPE handles any language, any word, any misspelling — unknown words are decomposed into known subword pieces. GPT models use BPE. Claude and GPT-4 use vocabularies of ~100K tokens.
WordPiece BERT's tokenisation variant. Similar to BPE but merges pairs that maximise the likelihood of the training data (likelihood-based, not frequency-based). Uses "##" prefix for continuation tokens: "unhappiness" → ["un", "##happi", "##ness"]. Vocabulary size typically 30,522 for BERT-base. Less common now — most modern models use BPE or its variant (tiktoken for GPT, SentencePiece for many others).
Vocabulary Size The number of unique tokens the model knows. BERT: 30,522. GPT-2: 50,257. GPT-4 / Claude: ~100,000. Larger vocabulary means fewer tokens per word (more efficient context window usage) but more embedding parameters (100K tokens × 4096 dims = 400M params just for embeddings). The vocabulary is fixed after pre-training — the model can't learn new tokens without retraining the tokeniser and embedding layer.
Scaling Laws The empirical observation (Kaplan et al., OpenAI 2020; Hoffmann et al., DeepMind 2022 — "Chinchilla") that model performance improves predictably as a power law with parameter count, dataset size, and compute. The Chinchilla paper found that many large models were undertrained: a 70B model trained on 1.4T tokens is better than a 280B model trained on 300B tokens. Optimal training: roughly 20 tokens of training data per parameter. This guides every major LLM training run — before you spend millions on compute, you can predict the improvement you'll get.
Emergent Abilities Capabilities that appear suddenly at large scale that are not present (or even slightly present) in smaller models. Below a certain scale: zero performance. At that scale: near-perfect performance. Examples: arithmetic with carry (addition), multi-step reasoning, in-context learning, chain-of-thought, code generation. These abilities were not explicitly trained — they emerged from scale. This is why small models sometimes can't do things that seem simple: the emergent threshold hasn't been crossed. GPT-3 (175B) was when many emergent abilities first appeared.
Why scale matters — the threshold analogy

A 1-parameter model can't do anything useful. A 1B-parameter model can complete sentences. A 7B model can follow instructions. A 70B model starts reasoning. A 175B model shows emergent abilities. The graph isn't smooth — it has phase transitions. Like how water doesn't "partially freeze" at 5°C — it stays liquid until exactly 0°C, then changes state. Emergent abilities in LLMs are phase transitions: the model either has them or it doesn't, and scale is the temperature control.

Part 2 · Fine-Tuning and Alignment
Block 06 · Section 2
Fine-Tuning & Alignment — From Text Predictor to Helpful Assistant
A pre-trained model completes text — it doesn't answer questions helpfully, refuse harmful requests, or behave safely. These three steps turn a raw model into a production assistant.
Fine-Tuning

From raw model to helpful assistant — three stages

A freshly pre-trained model is impressive but dangerous. Ask it "How do I make a bomb?" — it will helpfully complete the text. Ask it "What's the capital of France?" — it might respond "What's the capital of France? That's a great question. The capital of Spain is Madrid, and..." (it completes patterns, not questions). Three training stages fix this.

Supervised Fine-Tuning (SFT) After pre-training, train the model on a dataset of (instruction, ideal response) pairs. "Summarise this article: [article]" → "[clean summary]". "Write a Python function to sort a list" → "[correct Python code]". "What is photosynthesis?" → "[accurate explanation]". This is standard supervised training: model predicts tokens of the ideal response, loss is backpropagated. The model starts to understand that its job is to answer, not just complete. Typically uses a dataset of 100K–1M carefully curated examples. InstructGPT was the first widely publicised SFT model.
Instruction Tuning A specific type of SFT using a very diverse set of instruction types — writing, coding, summarising, translating, reasoning, roleplay, math, safety refusals, and more. The diversity is what makes the model generalisable: a model fine-tuned only on writing prompts won't know how to follow coding instructions. Instruction tuning gives the model a general "follow instruction" capability that transfers to novel instruction types it hasn't explicitly seen. The FLAN models (Google) and Alpaca (Stanford) pioneered this approach with open datasets.
RLHF — Reinforcement Learning from Human Feedback The technique that made ChatGPT feel dramatically better than GPT-3 base. Three sub-stages: (1) SFT on demonstrations as above. (2) Reward Model training: collect pairs of model outputs for the same prompt, have human raters rank which is better, train a smaller model (the reward model) to predict human preference scores. (3) RL fine-tuning: use PPO (Proximal Policy Optimisation) to update the main model, using the reward model's scores as the reward signal. The model learns to generate outputs humans rate highly. Anthropic's variant is Constitutional AI (CAI) — uses AI feedback instead of human feedback for the RL stage, with a written "constitution" of values.
The alignment tax

RLHF makes models more helpful and safer, but it slightly reduces raw capability scores on certain benchmarks — the model sacrifices some completion quality to add helpfulness and safety. This trade-off is called the "alignment tax." It's considered worth paying — a model that follows instructions correctly and refuses to help with harm is far more useful in practice than a model that's slightly better at predicting tokens from a benchmark dataset. Modern alignment research is focused on reducing this tax while keeping the safety improvements.

Part 3 · SLM vs LLM
Block 06 · Section 3

SLM vs LLM — size is a design decision

Parameter count is not just a measure of capability — it determines where the model runs, how much it costs, and what tasks it can tackle. The right model size is the one that solves your problem at the acceptable cost and latency. Bigger isn't always better.

Small Language Models
1B – 7B parameters
  • Runs on consumer GPU (RTX 4090) or laptop
  • Fast inference — low latency responses
  • Cheap to run — fraction of LLM cost
  • Good for focused, well-defined tasks
  • Can run locally — full data privacy
  • Weaker at multi-step reasoning
  • Fewer emergent capabilities
Examples: Phi-3 (3.8B) · Gemma 2B · Mistral 7B · Llama 3.2 3B
Large Language Models
70B – 1T+ parameters
  • Requires GPU clusters or cloud API
  • Slower and more expensive per token
  • Broad general capability across all tasks
  • Strong multi-step reasoning and planning
  • Emergent abilities present and reliable
  • Better instruction following on complex prompts
  • Powers most production AI agents today
Examples: GPT-4o · Claude 3.5 Sonnet · Gemini 1.5 Pro · LLaMA 3 70B
The practical decision framework

Use an SLM when: the task is well-defined and narrow (sentiment classification, simple extraction, document tagging), latency is critical, data must stay on-premises, or cost at scale makes LLMs infeasible. Use an LLM when: the task requires multi-step reasoning, complex instruction following, broad knowledge, or writing quality matters. A common production pattern: use a small model to route and classify requests, a large model to handle complex queries that fail the small model's confidence threshold.

Part 4 · Inference and Generation
Block 06 · Section 4
Inference & Generation — Controlling What the Model Produces
The model outputs a probability distribution over all tokens at each step. Decoding strategies decide how to turn those probabilities into actual text. Four strategies. Three key parameters.
Inference

Decoding strategies — four ways to pick the next token

At every generation step, the model outputs a probability distribution over its entire vocabulary (~50K–100K tokens). The decoding strategy decides how to select one token from that distribution. The choice dramatically affects output quality and diversity.

Greedy Decoding
Always pick the highest-prob token
At each step, take the single token with the highest probability. Fast, deterministic — same prompt always gives the same output. But often produces repetitive, boring text — the model gets stuck looping on high-probability completions.
Use for: Factual Q&A where one correct answer exists. Never for creative writing.
Beam Search
Keep top-B sequences in parallel
Instead of committing to one token, keep the top B most likely partial sequences ("beams") at each step. After generation, select the sequence with the highest overall probability. Better than greedy but still deterministic and can still be repetitive.
Use for: Machine translation, summarisation where coherence matters more than diversity.
Top-k Sampling
Sample from top K tokens only
Filter to only the K most likely tokens, then sample randomly from those K. k=50 means: consider only the 50 most likely words, pick one probabilistically. Adds diversity. Problem: k=50 might include rare tokens when distribution is sharp, or miss good tokens when it's flat.
Use for: Creative writing, chatbots. k=40–50 is a common default.
Top-p (Nucleus) Sampling
Sample from tokens covering p% of probability mass
Dynamically select the smallest set of tokens whose cumulative probability exceeds p (e.g. 0.9). When the distribution is sharp, this selects few tokens. When it's flat, it selects many. Adapts to context. Considered superior to top-k for most tasks.
Use for: General text generation. p=0.9 is the standard default. Claude uses this.

Temperature — how it works mathematically Temperature T divides all logits (raw pre-softmax scores) before the softmax: logit_scaled = logit / T. At T=1: logits unchanged, standard distribution. At T→0: the highest logit becomes enormous relative to others after division → softmax is near-1 for the top token → greedy. At T=2: all logits are halved → differences between probabilities shrink → flatter distribution → more randomness. Temperature does not change which token is most likely — only how much more likely the top choice is versus alternatives.
Context Window The maximum number of tokens the model can process in one call — the "memory" of a single conversation or task. Includes both the input (prompt + conversation history + documents) and the output (the model's response). GPT-2: 1,024 tokens. GPT-3: 4,096. GPT-4-turbo: 128,000. Claude 3.5 Sonnet: 200,000. Gemini 1.5 Pro: 1,000,000. The context window is not free — attention computation scales as O(n²) with sequence length, so very long contexts are expensive. When the context is full, the oldest tokens are dropped (unless you implement summarisation or memory management).
Max Tokens The maximum number of tokens the model will generate in its response. Hard stop — the model simply stops generating when it hits this limit, even if the response is incomplete. Set this based on the expected response length: 100 for short answers, 1000 for essays, 4096 for detailed code, 8192 for long documents. Setting it too low truncates responses. Setting it too high wastes cost if responses are short. Most APIs also have a global context window limit that counts both input and output tokens together.
Part 5 · Prompting Intro
Block 06 · Section 5

Prompting intro — three ways to talk to an LLM

Before diving deep into Prompt Engineering (Block 08 in your roadmap), here are the three fundamental prompting patterns every LLM user needs to know. These are the building blocks everything else is built from.

PatternWhat it isWhen to useExample
Zero-shot Just ask — no examples or demonstrations. The model uses its pre-training knowledge directly. Simple, common tasks where the model has seen enough examples during training. "Translate to French: Hello, how are you?"
Few-shot Give 2–5 input/output examples in the prompt before your actual query. The model learns the pattern from examples. When output format matters, when task is unusual, when you need consistent style. "Positive: I love this! → POS\nNegative: Terrible. → NEG\nReview: Pretty good. → ?"
System prompt A persistent instruction that sets the model's role, persona, constraints, and context for the entire conversation. Every production application. Defines who the model is and what it should/shouldn't do. "You are a helpful customer support agent for Acme Corp. Only answer questions about our products. Be concise."

Zero-shot, few-shot, and system prompts are the entry points. Block 08 (Prompt Engineering) covers Chain-of-Thought, ReAct, self-consistency, prompt chaining, and every advanced technique that makes LLMs genuinely reliable in production. For now, know these three — they're the vocabulary of every LLM interaction you'll have.

LLM Embeddings & Vector Search — Text to Meaning to Retrieval
LLMs can convert any text into a high-dimensional vector that captures its semantic meaning. These vectors power semantic search, document retrieval, and the entire RAG architecture.
Vector dims: ~1536 (OpenAI text-embedding-3-small)
Tools: FAISS · Chroma · Pinecone
Block 06 · Section 6

LLM Embeddings — text into meaning space

Beyond generating text, LLMs can also function as embedding models — converting any text (word, sentence, paragraph, document) into a dense vector of numbers that captures semantic meaning. These vectors are the backbone of every modern search system and every RAG pipeline.

ConceptWhat it means in practice
Text → Embedding Vector Any text input → a fixed-length list of ~1536 numbers representing its meaning. "The cat sat on the mat" → [0.023, −0.418, 0.891, …]. Semantically similar texts produce numerically similar vectors.
Cosine Similarity Measure how similar two vectors are by computing the cosine of the angle between them. 1.0 = identical meaning, 0.0 = unrelated, −1.0 = opposite. The standard distance metric for all embedding search.
Semantic Search Find documents that mean the same thing as a query, not just documents that share keywords. "How do I sort a list in Python?" finds "The sorted() function arranges elements in order" even though "sorted", "function", and "order" don't appear in the query.
Vector Database A database optimised for storing millions of embedding vectors and finding the nearest neighbours to a query vector extremely fast. Not a traditional SQL database — uses approximate nearest neighbour (ANN) algorithms like HNSW or IVF.
FAISS / Chroma / Pinecone FAISS (Facebook AI Similarity Search): open-source, runs locally, extremely fast, best for large-scale production. Chroma: developer-friendly, easy setup, good for prototyping. Pinecone: managed cloud service, no infrastructure to manage, production-ready with scaling built-in.
RAG — Retrieval-Augmented Generation The pattern: embed all your documents → store in vector DB → at query time, embed the user's question → find the top-k most similar document chunks → inject those chunks into the LLM's context → LLM answers using the retrieved facts as grounding. Solves the knowledge cutoff problem and reduces hallucinations. Full deep-dive in Block 10 (RAG).
Connects directly to Agentic AI, RAG, and Context Engineering

Embeddings + vector search are the retrieval infrastructure of every production AI agent. When an agent needs to answer a question about your company's internal documents, it doesn't re-read all documents every time — it embeds the query, finds the top 5 relevant chunks in milliseconds, and feeds only those to the LLM. This makes agents both accurate (grounded in real data) and efficient (not paying to process millions of tokens on every query). The full technical implementation is Block 10 (RAG).

Part 7 · How Generation Works
Block 07 · Section 1

How generation works — one token at a time

Generative AI is the application layer of everything you've learned. The Transformer architecture from Block 05, trained and aligned via the Block 06 process, now takes a prompt and creates something new — text, images, audio, or combinations of all three.

Autoregressive Generation The core generation mechanism for text LLMs. Predict one token, append it to the sequence, predict the next token from the extended sequence, repeat. "Write me a poem about stars" → model predicts "The" → appends to get "Write me a poem about stars\nThe" → predicts "stars" → appends → predicts "above" → and so on until a stop condition is met. Each step is a full forward pass through the entire Transformer. This is why generating long responses takes more time than short ones — it scales linearly with output length.
Stop Sequences Tokens or strings that tell the model to stop generating immediately when encountered. Common examples: "\n\n" (double newline — stop after a paragraph), "Human:" or "User:" (stop before the next conversation turn in chat systems), a custom delimiter like "###END###". Without stop sequences, the model generates until max_tokens is reached. Well-designed stop sequences prevent runaway generation and keep outputs clean and correctly delimited.
The full generation pipeline

User prompt → Tokenise (text → token IDs) → Embed (IDs → vectors) → Add positional encoding → Pass through N Transformer blocks (attention + FFN × N) → Final layer norm → Project to vocabulary (linear layer, shape d_model × vocab_size) → Apply temperature → Apply top-p or top-k mask → Softmax → Sample one token → Detokenise (token ID → text fragment) → Append to output → Repeat from "Embed" step. Every token generation goes through this full pipeline. For a 200-token response from a 96-layer model: 200 × 96 × (attention + FFN operations) = enormous amount of computation, all happening on GPU clusters in seconds.

Image generation — deeper

Diffusion Models — learning to un-noise

Image generation uses a completely different architecture from text LLMs — diffusion models, not Transformers (though modern image models like DALL-E 3 do incorporate Transformer components). The core insight is elegant: instead of learning to generate images directly, teach the model to reverse a well-defined destruction process.

Forward diffusion is a fixed mathematical process — gradually add Gaussian noise according to a schedule. The model only needs to learn one thing: given a noisy image at time t, what noise was added? Train it to predict the noise. At inference, start from pure random noise and run the reverse process ~50 steps, guided at each step by the text prompt's CLIP embedding.
CLIP — Contrastive Language-Image Pre-training OpenAI's model that jointly trains a text encoder and image encoder so that matching image-text pairs have similar embeddings in shared vector space. "A photo of a red cat" → text embedding. [photo of a red cat] → image embedding. Trained on 400M image-text pairs from the internet. CLIP bridges text and images: the text embedding of a generation prompt is used to condition the diffusion denoising process, steering it toward images that match the text. CLIP is why "photorealistic photo of a dragon flying over Mumbai at sunset" actually generates something recognisable — the text embedding guides every denoising step.
Why diffusion models, not Transformers, for images

Text generation works well autoregressively (one token at a time) because language has a natural left-to-right order. Images don't have that order — top-left to bottom-right pixel generation produces blurry, incoherent images. Diffusion models generate the entire image simultaneously through iterative refinement, preserving global coherence. Recent work (like DALL-E 3 and Stable Diffusion 3) does incorporate Transformer components for image understanding, but the core generation mechanism remains diffusion.

Evaluation — deeper

How we measure whether AI output is good

BLEU — Bilingual Evaluation Understudy Measure overlap between generated text and a human reference translation. Count n-gram matches (1-gram, 2-gram, 3-gram, 4-gram) between generated and reference text. Average across n-gram sizes. Score: 0.0 (no overlap) to 1.0 (perfect match). Widely used for machine translation. Limitation: doesn't account for paraphrases — "The vehicle approached quickly" vs "The car came fast" are semantically similar but BLEU would score them low.
ROUGE — Recall-Oriented Understudy for Gisting Evaluation Measures how much of the reference text appears in the generated text (recall-focused, unlike BLEU which is precision-focused). ROUGE-1: unigram recall. ROUGE-2: bigram recall. ROUGE-L: longest common subsequence. Standard for evaluating text summarisation — how much of the important content from the source did the summary capture? Still can't handle paraphrases or judge quality of reasoning.
FID Score — Fréchet Inception Distance Measure the quality and diversity of generated images by comparing their statistical distribution against real images. Pass both generated and real images through an Inception network, extract feature vectors, compare the mean and covariance matrices of both distributions. Lower FID = generated images more closely match the statistical distribution of real images = better quality and diversity. Unlike human evaluation, FID is automatic and scalable. Widely used to benchmark image generation models: Stable Diffusion FID ≈ 7, DALL-E 3 FID ≈ 5 (lower is better).
LLM-as-Judge Use a capable LLM (GPT-4, Claude) to evaluate the output of another LLM. Given a prompt and a response, ask the judge: "Rate this response from 1-10 for helpfulness, accuracy, and coherence. Explain your reasoning." Scales to millions of evaluations automatically. Correlates well with human judgment on most tasks. Used heavily in model development: run 10,000 prompt-response pairs through an LLM judge to get aggregate quality scores for a model checkpoint. Main risk: the judge's biases become your evaluation biases — GPT-4 rated as judge tends to prefer GPT-4-style responses.
Part 9 · Multimodal AI
Block 07 · The Current Frontier
Multimodal AI — Text + Image + Audio + Video in One Model
Instead of separate models for each modality, modern frontier models process all data types simultaneously. The direction every major AI lab is heading — because the real world is multimodal.
Multimodal AI

Multimodal — one model, every sense

The history of AI has been specialisation: image models for images, speech models for speech, text models for text. Multimodal AI breaks that pattern — one model that can simultaneously understand and generate across all data types. GPT-4o can listen to you speak, look at the photo you took, and respond in both text and voice. Gemini 1.5 Pro can watch a video and answer questions about it. This isn't just convenient — it's architecturally more powerful because modalities inform each other.

Why multimodal is better than single-modal

When a radiologist reads an X-ray, they also read the patient's notes, ask follow-up questions, look at previous scans, and integrate all of it into one diagnosis. A text-only model can't see the X-ray. An image-only model can't read the notes. A multimodal model can do all of it simultaneously — just like the radiologist. Real-world intelligence is inherently multimodal. Specialised single-modality models are a historical artefact of limited compute and architectural limitations that frontier models are now overcoming.

Vision Encoder Converts an image into a sequence of tokens the LLM can attend to. Typically a Vision Transformer (ViT) that splits the image into patches (e.g. 16×16 pixel patches), projects each patch into a vector, and produces a sequence of patch tokens. A 224×224 image becomes 196 patches (14×14 grid), each mapped to a 768-dimensional vector — a sequence of 196 tokens the LLM can attend to alongside the text tokens. The vision encoder's parameters are pre-trained separately (often on CLIP) and then combined with the LLM.
Audio Encoder Converts audio waveforms into tokens. Typically splits audio into short overlapping frames (e.g. 25ms windows), computes spectral features, and projects them into token vectors. Whisper-style encoders use a CNN-Transformer stack to convert mel spectrograms into token sequences. GPT-4o processes audio natively end-to-end (audio waveform → tokens → LLM → text + audio output) without a separate transcription step, enabling real-time voice conversations with genuine emotional understanding from tone and pacing.
Models to know for multimodal AI

GPT-4o — OpenAI's "omni" model. Text + image + audio natively in one model. Real-time voice conversation. Can sing, detect emotions in speech, describe images. Gemini 1.5 Pro — Google's leading multimodal. Text + image + video + audio. 1M token context window. Can analyse an entire film. Claude 3.5 Sonnet — Anthropic's model. Text + image understanding. Excellent at document analysis, chart reading, code screenshots. LLaVA — open-source vision-language model. CLIP vision encoder + LLaMA LLM. Run locally. Whisper + GPT pipeline — Whisper for speech-to-text, GPT for understanding/generation. The legacy approach before native multimodal models became standard.

Cell 2 — Decoding Strategies: Greedy vs Top-p vs Temperature
import torch
import torch.nn.functional as F
import numpy as np
torch.manual_seed(42)

# ── Simulate a token probability distribution ──────────────────
# Pretend the model has computed logits for the next token over a small 10-word vocab
vocab = ["cat", "dog", "mat", "sat", "the", "a", "big", "tiny", "lazy", "jumped"]
logits = torch.tensor([3.2, 1.8, 2.9, 0.5, 1.2, 0.8, 0.3, 0.1, 0.2, 0.4])
# logits: raw model outputs before softmax. Higher = model thinks this word is more likely.

print("=== Token Probability Distribution ===\n")
probs_default = F.softmax(logits, dim=-1)
print(f"{'Token':10} {'Logit':>8} {'Prob (T=1.0)':>12} {'Bar'")
print("-" * 45)
for tok, log, prob in zip(vocab, logits, probs_default):
    bar = '█' * int(prob.item() * 50)
    print(f"{tok:10} {log.item():>8.1f} {prob.item():>12.4f} {bar}")

# ── Effect of Temperature ──────────────────────────────────────
print("\n\n=== Temperature Effect on Probabilities ===\n")
print(f"{'Temperature':14} {'cat (was top)':>15} {'dog':>8} {'mat':>8} {'Effect'")
print("-" * 60)
for temp in [0.1, 0.5, 1.0, 1.5, 2.0]:
    scaled = F.softmax(logits / temp, dim=-1)
    # logits / temp: divide all logits by temperature before softmax
    # low T: top logit gets relatively huge → near-1 probability
    # high T: all logits shrink proportionally → flatter distribution
    effect = "near-greedy" if temp < 0.3 else ("creative" if temp > 1.2 else "balanced")
    print(f"T = {temp:9.1f}  {scaled[0].item():>15.4f} {scaled[1].item():>8.4f} {scaled[2].item():>8.4f}  {effect}")

# ── Top-p (Nucleus) Sampling ───────────────────────────────────
print("\n\n=== Top-p (Nucleus) Sampling (p=0.9) ===\n")
probs_sorted, sorted_idx = torch.sort(probs_default, descending=True)
# sort probabilities highest to lowest, keeping original indices
cumulative = torch.cumsum(probs_sorted, dim=-1)
# cumsum: running sum — cumulative[i] = sum of top-i probabilities

p = 0.90
nucleus_mask = cumulative <= p
nucleus_mask[0] = True   # always include the top token
nucleus_tokens = [vocab[sorted_idx[i]] for i in range(len(vocab)) if nucleus_mask[i]]
nucleus_probs  = probs_sorted[nucleus_mask]

print(f"Top-p={p}: Sample from these tokens (covering {nucleus_probs.sum():.3f} of probability):")
for tok, prob, cum in zip(nucleus_tokens, nucleus_probs, cumulative[nucleus_mask]):
    print(f"  {tok:8} prob={prob:.4f}  cumulative={cum:.4f}")
print(f"\nTop-p EXCLUDED: {[vocab[sorted_idx[i]] for i in range(len(vocab)) if not nucleus_mask[i]]}")
print("(low-probability tokens are excluded from sampling — prevents weird word choices)")

# Using sentence-transformers for local embeddings (no API key needed)
import subprocess
subprocess.run(["pip", "install", "sentence-transformers", "faiss-cpu", "-q"])
from sentence_transformers import SentenceTransformer
import faiss           # FAISS: Facebook AI Similarity Search — fast vector database
import numpy as np

model_emb = SentenceTransformer('all-MiniLM-L6-v2')
print("✅ Embedding model loaded\n")

# ── Create a mini document corpus ─────────────────────────────
documents = [
    "Python is a high-level programming language popular for AI and data science.",
    "The Transformer architecture uses self-attention to process sequences in parallel.",
    "Chennai is a coastal city in Tamil Nadu, India, known for its culture and food.",
    "Gradient descent optimises neural network weights by minimising the loss function.",
    "RLHF trains language models to be helpful using human preference feedback.",
    "The Indian Ocean is the third largest ocean in the world.",
    "Fine-tuning adapts a pre-trained model to a specific domain or task.",
    "DALL-E generates images from text descriptions using a diffusion model.",
]

# ── Step 1: Embed all documents (indexing phase) ──────────────
doc_embeddings = model_emb.encode(documents)
# encode(): returns numpy array shape (n_docs, embedding_dim)
doc_embeddings = doc_embeddings.astype(np.float32)
# FAISS requires float32

# ── Step 2: Build FAISS index (vector database) ───────────────
d = doc_embeddings.shape[1]   # embedding dimension (384 for MiniLM)
index = faiss.IndexFlatIP(d)   # IndexFlatIP: Inner Product search (= cosine when normalised)

faiss.normalize_L2(doc_embeddings)
# normalize_L2: convert embeddings to unit vectors so inner product = cosine similarity
index.add(doc_embeddings)
# add(): insert all document vectors into the index
print(f"Vector database built: {index.ntotal} documents, {d} dimensions each")

# ── Step 3: Semantic search (retrieval phase) ─────────────────
queries = [
    "How do LLMs learn from human feedback?",
    "What programming language is best for machine learning?",
    "How are images generated by AI?",
]

print("\n=== Semantic Search Results ===\n")
for query in queries:
    q_emb = model_emb.encode([query]).astype(np.float32)
    faiss.normalize_L2(q_emb)        # normalise query too
    scores, indices = index.search(q_emb, k=2)
    # search: find top-k most similar document vectors
    # returns scores (cosine similarity) and indices (which document)
    print(f"Query: '{query}'")
    for rank, (score, idx) in enumerate(zip(scores[0], indices[0]), 1):
        print(f"  Rank {rank} (score={score:.4f}): {documents[idx]}")
    print()

import subprocess
subprocess.run(["pip", "install", "nltk", "rouge-score", "-q"])
from nltk.translate.bleu_score import sentence_bleu, SmoothingFunction
from rouge_score import rouge_scorer
import nltk; nltk.download('punkt', quiet=True)

# ── BLEU — measures translation/generation quality ────────────
print("=== BLEU Score ===\n")
reference = "The cat sat on the warm mat near the window"
candidates = [
    "The cat sat on the warm mat near the window",  # perfect
    "The cat sat on the mat",                        # partial match
    "A dog stood on a cold floor beside the door",   # poor match
    "The feline rested on the warm rug by the glass", # paraphrase — BLEU weakness!
]

ref_tokens = reference.lower().split()
smooth = SmoothingFunction().method1
# SmoothingFunction: prevents zero BLEU for short sentences with no n-gram matches

for cand in candidates:
    cand_tokens = cand.lower().split()
    score = sentence_bleu([ref_tokens], cand_tokens, smoothing_function=smooth)
    # sentence_bleu([refs], candidate): first arg is list of references (can have multiple)
    # returns BLEU-4 by default: geometric mean of 1,2,3,4-gram precision
    print(f"BLEU = {score:.3f} | '{cand}'")

# ── ROUGE — measures summarisation quality ────────────────────
print("\n\n=== ROUGE Score ===\n")
article = """Machine learning is a branch of artificial intelligence. It enables systems 
to learn from data and improve over time. Neural networks are a key technique in 
machine learning that simulate how the brain works."""
summaries = {
    "Good summary":    "Machine learning enables systems to learn from data using neural networks.",
    "Partial summary": "AI systems learn from data.",
    "Off-topic":       "The weather in Chennai is warm and humid today.",
}

# rouge1: unigram overlap, rouge2: bigram overlap, rougeL: longest common subsequence
# use_stemmer=True: "learns" and "learning" count as the same word

print(f"{'Summary type':20} {'R1-F1':>8} {'R2-F1':>8} {'RL-F1':>8}")
print("-" * 50)
for name, summary in summaries.items():
    scores = scorer.score(article, summary)
    # score(reference, generated): returns dict of RougeScore objects with precision, recall, fmeasure
    print(f"{name:20} {scores['rouge1'].fmeasure:>8.3f} {scores['rouge2'].fmeasure:>8.3f} {scores['rougeL'].fmeasure:>8.3f}")

print("\nNote: Both BLEU and ROUGE struggle with paraphrases.")
print("LLM-as-judge handles paraphrases correctly — that's why it's now dominant.")

Connection to AI Agents
Blocks 06 + 07 — inside every agent you'll build

Language models are the brain. Generative AI is the voice. Both are inside every agent.

When you call the Claude or GPT-4 API in your agent, you're invoking the entire Block 06 stack: a decoder-only Transformer (Block 05), pre-trained on trillions of tokens via next-token prediction, aligned through SFT and RLHF, generating tokens via top-p sampling at temperature ~1.0, bounded by a 200K token context window. Every API parameter — model, max_tokens, temperature, system — maps directly to the concepts you just learned.

The decoding strategies you control with temperature and top-p matter for agents. Tool-calling and structured JSON output = low temperature (0.0–0.2), deterministic, reproducible. Reasoning and plan generation = medium temperature (0.5–0.7), coherent but flexible. Brainstorming and creative fallbacks = high temperature (0.9–1.2), diverse. Choosing the wrong temperature causes agent failures that look mysterious but are actually just sampling strategy mismatches.

Multimodal capability directly expands what agents can perceive. A customer service agent that can only read text is blind to product photos. An agent that can process images can read screenshots, receipts, charts, and diagrams. An agent with audio input can handle voice interactions natively. As you build agents in Blocks 09–13, multimodal input is what determines whether your agent can handle real-world messy data or only clean text.

LLM embeddings and vector search (Section 6 of Block 06) are the foundation of every RAG system — which is Block 10 in your roadmap. The full RAG pipeline adds prompt construction and LLM generation on top of exactly that retrieval foundation.
