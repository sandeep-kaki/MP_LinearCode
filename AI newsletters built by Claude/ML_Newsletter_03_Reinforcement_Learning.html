From the editor — the final issue

We've come a long way. Issue 01 was Supervised Learning — machines that learn from labelled examples. Issue 02 was Unsupervised Learning — machines that discover hidden structure in raw data with no labels at all.

This is Issue 03. And Reinforcement Learning is the strangest, most fascinating type of all.

It has no dataset. No examples of correct answers. No teacher giving feedback on every step. There is only an agent dropped into an environment — and the agent has to figure everything out by interacting with that environment, making mistakes, receiving sparse rewards, and slowly — across thousands of attempts — building up wisdom about what actions lead to good outcomes.

This is how AlphaGo learned to beat the world champion at Go. How OpenAI Five beat professional Dota 2 players. And most importantly for us — it is the core algorithm that makes Claude, ChatGPT, and Gemini actually helpful and safe. That's called RLHF, and by the end of this issue, you'll understand it deeply.

Part 1 · The Mental Model Shift
Everything changes here

Three types of learning — one final shift

Before we go deep, let's place RL in the context of everything you've learned:

Supervised Learning
  • You have labelled data
  • Correct answers exist
  • Teacher guides every step
  • Learn mapping: input → label
  • Example: classify images
Unsupervised Learning
  • No labels at all
  • No correct answers
  • No teacher — find structure
  • Discover: groupings, patterns
  • Example: cluster customers
Reinforcement Learning
  • No dataset at all
  • No correct answers anywhere
  • Learn from consequences
  • Maximise: cumulative reward
  • Example: learn to play chess

The crucial difference: in supervised and unsupervised learning, you have a static dataset you can study. In RL, there is no dataset. The agent creates its own data by interacting with the environment. Every action it takes, every reward it receives — that's the training data, generated on the fly, in real time.

Reinforcement learning is the only type of machine learning that learns the way a human child learns — not from a textbook, but from touching, failing, adjusting, and trying again.

The perfect analogy

You've just started learning to ride a bicycle. Nobody can give you a dataset of "correct body positions at every millisecond." No textbook can transfer the skill. You just get on. You wobble. You fall. Your brain registers: that lean was too far — bad outcome. You get up. You try again. Slightly different lean. Better. You repeat this thousands of times over days. Gradually, without anyone explicitly programming you, your brain builds a complete policy for balancing on a bicycle. That is reinforcement learning. The bicycle is the environment. Your body is the agent. Falling is negative reward. Staying upright is positive reward. The skill you build is the policy.

Part 2 · The Six Core Concepts
The vocabulary you must own

Six words that explain all of RL

Every RL system in existence — from Q-learning to the algorithm inside ChatGPT — is built on six concepts. Learn these once, understand them deeply, and RL will never confuse you again.

🤖
Agent
The learner and decision-maker. The entity that takes actions and receives consequences. It has no predefined knowledge of what's good or bad — it discovers this through experience.
In a chess game: the AI player. In RLHF: the language model itself.
🌍
Environment
Everything the agent interacts with. The environment receives the agent's actions, updates its own state, and returns a new state and a reward signal back to the agent.
In a chess game: the board and rules. In RLHF: the human evaluators judging responses.
👁️
State (s)
A snapshot of the environment at a specific moment. What the agent can observe right now. The agent makes decisions based on the current state.
In a maze: which cell the agent is currently in. In chess: the current positions of all pieces on the board.
🎮
Action (a)
Something the agent can do. A choice made in a given state. The set of all possible actions is called the action space. The agent's job is to choose the best action in each state.
In a maze: move Up, Down, Left, or Right. In RLHF: generating a specific response to a user prompt.
🏆
Reward (r)
A scalar number the environment gives the agent after each action. Positive = good. Negative = bad. Sparse (rare) rewards are the norm in real RL — you don't get feedback on every step, only at key moments.
Maze: +10 for reaching goal, -1 for each step (encourages efficiency), -5 for hitting a wall.
📋
Policy (π)
The agent's strategy. A function that maps every state to an action. The policy is what we're trying to learn. At the start it's random. After training it's the best possible strategy.
π(state = cell_3) = "move Right". A good policy always picks the action that leads to maximum cumulative reward.
The one extra concept — Return (G)

Reward is the immediate feedback from one action. Return is the total accumulated reward from now until the end of the episode. RL doesn't just maximise the next reward — it maximises the sum of all future rewards. This is why RL agents can make short-term sacrifices (accepting a small negative reward now) to get a much bigger reward later. Return is discounted: rewards further in the future are worth slightly less than immediate ones. The discount factor γ (gamma) controls this — typically 0.9 to 0.99.

Part 3 · The RL Loop
How an agent actually learns

The learning loop — four steps that repeat forever

The entire RL learning process is one loop, repeated thousands of times. Understanding this loop is understanding all of RL:

  • 01
    Observe state. The agent looks at the current environment and records its state s. This is everything it can see right now.
  • 02
    Choose action. Based on the current state and its current policy, the agent picks an action a. Early in training the policy is random. Later it becomes smarter.
  • 03
    Execute action, receive feedback. The agent takes action a. The environment updates to a new state s' and gives the agent a reward r. Did this action help or hurt?
  • 04
    Learn and update. The agent updates its internal knowledge using the experience (s, a, r, s'). This is where Q-learning happens. The policy improves slightly.
  • 05
    Repeat. Set s = s' (the new state becomes the current state). Go back to step 1. Repeat until the episode ends (reached goal, fell off cliff, game over). Then start a new episode.
Episode vs Step

A step is one iteration of the loop (one action taken). An episode is a full sequence of steps from start to finish (start of game to end). Training happens across thousands of episodes. Each episode the agent starts fresh but carries its updated policy from all previous episodes. Over time the policy gets better and episodes get longer (the agent survives longer, or reaches the goal faster).

Part 4 · Q-Learning
The core algorithm

Q-Learning — teaching an agent what actions are worth

Q-Learning is the fundamental RL algorithm. The idea: instead of directly learning the policy, learn a value function — a function that estimates how good each action is in each state. Then the policy is simple: always take the highest-value action.

This value function is called the Q-function (Q stands for "Quality"):

The Q-function Q(state, action) = expected total future reward if I take this action in this state and then play optimally

If you know Q for every (state, action) pair — you have your policy. Just look up the action with the highest Q-value in the current state. Done.

The Q-Learning update rule — the heart of the algorithm

But how do you learn Q? You start with all zeros (no knowledge) and update it every time you take an action:

Bellman Equation — Q update rule Q(s, a) ← Q(s, a) + α × [ r + γ × max Q(s', a') − Q(s, a) ]

Break this down word by word:

SymbolNameWhat it means
Q(s, a)Current Q-valueWhat we currently think this action is worth in this state
α (alpha)Learning rateHow much to update on each step — same as in gradient descent (0.01–0.5)
rReward receivedThe actual reward the environment gave for taking action a in state s
γ (gamma)Discount factorHow much future rewards are worth vs immediate rewards (0.9–0.99)
max Q(s', a')Best future Q-valueThe Q-value of the best action available in the NEXT state s'
[ r + γ × max Q(s',a') − Q(s,a) ]TD ErrorHow wrong our current estimate was. This is the correction signal.
The intuition in plain English

"What I currently think this action is worth" + learning_rate × ("what I now know it was actually worth" − "what I thought it was worth") = "my updated belief." It's exactly like gradient descent — you have an estimate, you see the truth, you move your estimate a little toward the truth. Repeat thousands of times. The Q-values converge to the correct values.

The Q-Table — RL's brain in a spreadsheet

For small environments, Q-values are stored in a table — a grid where rows are states and columns are actions. After training, every cell contains the expected future reward of taking that action in that state.

Now here's what the Q-Table looks like for a few states — before training (all zeros) vs after training (Q-learning has converged):

Q-Table after training — a few key states shown
State (cell) Q(Up) Q(Down) Q(Left) Q(Right) Best Action
S (0,0) Start -0.10 7.29 -0.10 -0.10 ↓ Down
(2,0) Mid-left -0.10 6.56 -0.10 1.23 ↓ Down

See what happened? The Q-table learned the entire optimal path through the maze — without anyone telling it the path. It discovered, purely through repeated trial-and-error and the Bellman update rule, that "going down" is the right move from the start, that the trap cell is dangerous (notice the large negative Q-values for moving toward it), and that from the cell next to the goal, "go right" is worth 9.90 in future rewards.

Why Q-learning works — the magic

Q-values propagate backwards from the goal. The cell next to the goal learns a high Q-value first (it's only one step away). Then the cells next to THAT cell update — their Q-values increase because they're near a high-Q state. This ripples all the way back to the starting cell. After enough episodes, every cell knows exactly how close it is to the goal and what direction to go. This backward propagation of value information is called temporal difference learning.

Part 6 · Exploration vs Exploitation
The central dilemma of RL

Exploration vs Exploitation — the dilemma every agent faces

Here's the core tension in all of reinforcement learning: should the agent use what it already knows works (exploitation), or try new things to discover if something better exists (exploration)?

The restaurant analogy

You know a restaurant you like. You go there every Friday (exploitation — you know it's good). But maybe there's an even better restaurant nearby you've never tried. If you never explore, you'll never find out. But if you always try new places, you never get to enjoy your favourite. The optimal strategy balances both — mostly go to your favourite, but occasionally try somewhere new. That's exactly what the epsilon-greedy algorithm does for RL agents.

Epsilon-Greedy — the standard solution

Epsilon-Greedy Policy With probability ε → explore (random action) | With probability (1−ε) → exploit (best Q-value action)
  • →
    Start with ε = 1.0 (100% exploration). The agent knows nothing, so random actions are just as good as anything else.
  • →
    Decay ε over time (e.g., multiply by 0.995 after each episode). As the agent learns, it should explore less and exploit more.
  • →
    Set a minimum ε (e.g., 0.01). Never drop to zero — always keep a tiny chance of exploration to handle environment changes.
Why this matters for AI agents you'll build

When you build an AI agent that uses tools — a web searcher, code executor, database querier — your agent faces the same dilemma. Does it use the tool that worked last time (exploit)? Or try a different combination of tools that might work better (explore)? Modern agent frameworks handle this with temperature settings on the LLM and beam-search style multi-path planning. It's the same concept — different implementation.

Part 7 · RLHF — The Big One
How Claude actually learns from humans

RLHF — Reinforcement Learning from Human Feedback

This is the most important section of the entire three-issue series. RLHF is the technique that turns a raw language model — which just predicts the next word, has no values, and might say harmful things — into a helpful, harmless, and honest assistant like Claude.

Understanding RLHF means understanding why AI assistants behave the way they do. Why Claude declines certain requests. Why ChatGPT's tone changed between versions. Why these models seem to have values. It all comes from RLHF.

The three-phase process

1
Pre-training — The Raw Model
Train a large transformer on trillions of tokens of internet text using standard next-token prediction. This model learns language, facts, and reasoning — but it's not helpful. Ask it "how do I bake a cake?" and it might just continue the text in a random direction. It has no concept of being an assistant. It knows what words follow what words. That's it.
Self-supervised learning
2
Supervised Fine-Tuning (SFT) — Teaching Behaviour
Human contractors write thousands of example (prompt → ideal response) pairs. "What is photosynthesis?" → [detailed, helpful, accurate answer]. The pre-trained model is fine-tuned on this dataset using standard supervised learning. Now it knows the format of helpful responses — but it still makes mistakes, is sometimes verbose, sometimes sycophantic, sometimes subtly wrong.
Supervised learning
3
Reward Model Training — Teaching Preference
Take the SFT model and generate multiple responses to the same prompt. Human raters rank these responses: "Response A was better than B, B was better than C." These human preference comparisons are used to train a separate neural network called the Reward Model. This model learns to predict "how much would a human prefer this response?" — it outputs a single score. This is supervised learning on preference data.
Supervised learning on preference pairs
4
RL Fine-Tuning — PPO + Reward Model
Now the RL part. The SFT model becomes the agent. Generating a response is the action. The user prompt is the state. The reward model scores the response — that score is the reward. An RL algorithm called PPO (Proximal Policy Optimisation) updates the LLM's weights to generate responses that receive higher reward model scores. Over thousands of iterations, the model learns to generate responses that humans prefer — more helpful, less harmful, better reasoned.
Reinforcement learning — PPO

The RL mapping for RLHF

RL ConceptWhat it maps to in RLHF
AgentThe language model (the LLM being trained)
EnvironmentThe user conversation + reward model evaluation
StateThe conversation history + current user prompt
ActionGenerating the next token (repeated until response is complete)
RewardScore from the reward model (how much a human would prefer this response)
PolicyThe LLM's weights — the probability distribution over next tokens
EpisodeOne full conversation turn: prompt → complete response generated
The critical limitation — reward hacking

The reward model is not perfect — it's a neural network approximating human preferences, trained on limited data. The LLM can learn to game the reward model: produce responses that score high on the reward model but aren't actually what humans would prefer. This is called reward hacking (a specific type of Goodhart's Law). The fix: PPO includes a KL-divergence penalty that prevents the model from drifting too far from the original SFT model. This keeps the RL fine-tuning stable. But it's an ongoing challenge in the field — which is why AI labs keep iterating on their RLHF processes.

RLHF variants you'll hear about

RLAIF (RL from AI Feedback) — instead of human raters, use another LLM to generate preference comparisons. Cheaper and faster, used heavily in production. DPO (Direct Preference Optimisation) — skips the reward model entirely and directly fine-tunes on preference pairs. More stable than PPO and increasingly popular. Constitutional AI (Anthropic's approach for Claude) — uses a set of written principles (a constitution) to guide RLAIF, making the values explicit and auditable.

Cell 1 — Build the Maze Environment
import numpy as np                    # fast array operations
import matplotlib.pyplot as plt      # plotting
import matplotlib.patches as mpatches # for legend patches in plots
from collections import defaultdict   # dict that auto-creates missing keys
import warnings
warnings.filterwarnings('ignore')

np.random.seed(42)    # reproducibility — same random results every run

# ── Define the 5×5 Maze ───────────────────────────────────────
# 0 = open cell, 1 = wall, 2 = trap (negative reward), 3 = goal
MAZE = np.array([
    [0, 0, 1, 0, 0],   # row 0: start at (0,0), wall at (0,2)
    [0, 1, 1, 0, 1],   # row 1: walls at (1,1),(1,2),(1,4)
    [0, 0, 0, 0, 0],   # row 2: all open
    [1, 0, 1, 2, 0],   # row 3: trap at (3,3), walls at (3,0),(3,2)
    [0, 0, 0, 0, 3],   # row 4: goal at (4,4)
])

START = (0, 0)    # agent always starts at top-left corner
GOAL  = (4, 4)    # agent must reach bottom-right corner

# ── Define actions as (row_change, col_change) pairs ──────────
ACTIONS = {
    0: (-1,  0),    # Up:    move one row up (row decreases by 1)
    1: ( 1,  0),    # Down:  move one row down
    2: ( 0, -1),    # Left:  move one column left
    3: ( 0,  1),    # Right: move one column right
}
ACTION_NAMES = {0: 'Up', 1: 'Down', 2: 'Left', 3: 'Right'}

def step(state, action):
    """Take one action from a state. Return (new_state, reward, done)."""
    row, col = state                              # unpack current position
    drow, dcol = ACTIONS[action]                  # get movement delta
    new_row, new_col = row + drow, col + dcol     # calculate new position

    # ── Check if new position is valid ────────────────────────────
    out_of_bounds = (new_row < 0 or new_row >= 5 or
                     new_col < 0 or new_col >= 5)
    # out_of_bounds: would moving here take us off the 5×5 grid?

    if out_of_bounds or MAZE[new_row, new_col] == 1:
        # hit a wall or went out of bounds — stay in place, penalise
        return state, -0.5, False

    new_state = (new_row, new_col)      # valid move — update position
    cell_type = MAZE[new_row, new_col]  # what type of cell is the new position?

    if cell_type == 3:     # reached the GOAL
        return new_state, +10.0, True   # big reward, episode ends (done=True)
    elif cell_type == 2:   # stepped on a TRAP
        return new_state, -5.0, False  # big penalty, but episode continues
    else:                   # normal open cell
        return new_state, -0.1, False  # small penalty per step (encourages efficiency)

print("✅ Maze environment ready")
print(f"Grid: {MAZE.shape}, Actions: {list(ACTION_NAMES.values())}")
print(f"Start: {START}, Goal: {GOAL}")
Cell 2 — Q-Learning Agent
# ── Q-Table: stores Q(state, action) for all state-action pairs
Q = defaultdict(lambda: np.zeros(4))
# defaultdict: auto-creates an entry for any new state we encounter
# lambda: np.zeros(4) = when a new state is seen, initialise Q-values to [0,0,0,0]
# 4 actions = 4 Q-values per state

# ── Hyperparameters ────────────────────────────────────────────
ALPHA   = 0.1      # learning rate: how fast we update Q-values (0=never, 1=instant)
GAMMA   = 0.95     # discount factor: how much we value future rewards vs immediate (0=greedy, 1=far-sighted)
EPSILON = 1.0      # starting exploration rate: 1.0 = 100% random at start
EPS_MIN = 0.01     # minimum exploration rate: always keep 1% random exploration
EPS_DECAY = 0.995 # multiply epsilon by this after each episode (slowly reduce randomness)
EPISODES = 3000   # total number of games to play (each game = one episode)
MAX_STEPS = 200   # max steps per episode (prevents infinite wandering)

# ── Tracking variables ─────────────────────────────────────────
rewards_per_episode = []   # total reward each episode (to track learning progress)
steps_per_episode   = []   # steps taken each episode (shorter = smarter agent)
epsilon_history     = []   # epsilon at each episode (to visualise exploration decay)

# ── The main Q-Learning training loop ─────────────────────────
for episode in range(EPISODES):

    state = START        # reset: agent starts fresh at the beginning every episode
    total_reward = 0    # accumulate total reward for this episode
    step_count = 0      # count steps taken this episode

    for _ in range(MAX_STEPS):    # run up to MAX_STEPS per episode

        # ── Step 1: Choose action using epsilon-greedy ────────────
        if np.random.random() < EPSILON:
            action = np.random.randint(4)    # EXPLORE: pick a random action (0,1,2,3)
        else:
            action = np.argmax(Q[state])     # EXPLOIT: pick the action with highest Q-value
            # np.argmax: returns index of the maximum value in the array

        # ── Step 2: Take action, observe result ───────────────────
        next_state, reward, done = step(state, action)
        # step() returns: new position, reward received, whether episode ended

        # ── Step 3: Q-Learning update (Bellman equation) ──────────
        best_next_q = np.max(Q[next_state])
        # best_next_q: the Q-value of the best action available from the next state
        # this is the "max Q(s', a')" term in the Bellman equation

        td_target = reward + GAMMA * best_next_q
        # td_target (Temporal Difference target): what we now believe this state-action is worth
        # = immediate reward + discounted value of best future action

        td_error = td_target - Q[state][action]
        # td_error: how wrong our previous Q-value estimate was
        # positive = we underestimated (increase Q), negative = overestimated (decrease Q)

        Q[state][action] += ALPHA * td_error
        # update Q-value: move current estimate toward td_target by a fraction (ALPHA)
        # same idea as gradient descent: small step in the right direction

        # ── Step 4: Update tracking variables ─────────────────────
        total_reward += reward    # accumulate reward for this step
        step_count   += 1
        state = next_state        # move to the new state

        if done: break    # if we reached the goal, end this episode early

    # ── After each episode: decay epsilon and record stats ────────
    EPSILON = max(EPS_MIN, EPSILON * EPS_DECAY)
    # max(EPS_MIN, ...): never let epsilon drop below EPS_MIN (keep some exploration)

    rewards_per_episode.append(total_reward)
    steps_per_episode.append(step_count)
    epsilon_history.append(EPSILON)

    if (episode + 1) % 500 == 0:    # print progress every 500 episodes
        recent_reward = np.mean(rewards_per_episode[-100:])
        # mean of last 100 episodes — smoother indicator of recent performance
        print(f"Episode {episode+1:4d} | Avg reward (last 100): {recent_reward:6.2f} | ε: {EPSILON:.3f}")

print("\n✅ Training complete!")

class MiniRewardModel:
    """A tiny reward model that scores LLM responses."""
    def score(self, response: str) -> float:
        """Score a response based on quality signals."""
        score = 0.0

        score += min(3.0, len(response.split()) * 0.1)
        # reward for length: more content is generally better (up to a point)

        helpful_words = ['because', 'therefore', 'example', 'however', 'step']
        score += sum(0.5 for w in helpful_words if w in response.lower())
        # reward for reasoning words: "because", "therefore" = structured thinking

        harmful_words = ['idk', 'whatever', 'maybe', 'dunno']
        score -= sum(1.0 for w in harmful_words if w in response.lower())
        # penalise vague/unhelpful responses

        return round(max(0.0, score), 2)    # clip to minimum 0

Part 9 · The Agentic AI Connection
The connection you've been building toward

RL is not just in the training — it's in the architecture of every agent you'll build

Every AI agent you build with LangChain, LangGraph, CrewAI, or Google ADK is, at its core, a reinforcement learning system. The agent (the LLM) observes a state (the conversation + tool results). It takes an action (generates text, calls a tool, makes a decision). It receives implicit feedback (the task succeeds or fails). It adjusts future behaviour based on that feedback.

The ReAct (Reason + Act) pattern — the most common agent architecture — is literally the RL loop: observe state → reason about actions → take action → observe result → update plan. The difference is the policy isn't updated via gradient descent in real-time. It was learned during RLHF training. But the structure is identical.

Multi-agent systems are even closer: an orchestrator agent assigns tasks to sub-agents, observes their outputs (rewards), and adjusts future task assignments. That's exactly Q-learning at the meta level. When you understand that, you understand why agent frameworks are designed the way they are — they're implementations of RL theory.
